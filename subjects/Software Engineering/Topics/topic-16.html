Topic 16

Software Reuse

Software products are expensive, and project managers are constantly seeking ways to cut development costs. Software reuse, the practice of using parts from previously developed software, is a promising approach. It's analogous to building electronic circuits with standard ICs and hardware components.

Benefits of Reuse: Beyond reducing development cost and time, reuse ensures higher quality as reusable components have typically already undergone rigorous testing and proven their reliability.

Component-based software development, which emphasizes assembling software from off-the-shelf components, is distinct from traditional development and is heavily reliant on effective reuse.

14.1 What Can Be Reused?

Many artifacts associated with software development can potentially be reused, including project plans and test plans. The prominent items that can be effectively reused are:

  • Requirements specification
  • Design
  • Code
  • Test cases
  • Knowledge

**Knowledge** is the most abstract development artifact that can be reused. While some knowledge reuse occurs automatically without conscious effort, effectively reusing potentially reusable development knowledge is difficult. It requires systematic extraction and documentation, which is often challenging.

14.2 Why Almost No Reuse So Far? (Challenges)

A common scenario in software development is the frustration of engineers realizing their current system is being built from scratch, without leveraging lessons or components from past, similar systems. This "reinvention of the wheel" often occurs because no one has the time to analyze similarities and exploit past work.

Even when organizations embark on reuse programs, they face significant challenges:

  • Creating components that are genuinely reusable across different applications is inherently difficult.
  • It's hard to anticipate the exact components that can be widely reused.
  • Even when reusable components are available, programmers often prefer to create their own due to difficulties in understanding or adapting existing ones.

**Observation:** Mathematical routines (e.g., `sin`, `cos`) are reused very successfully. This is because they are well-understood (everyone knows what `cosine` means), have a small, standardized interface (e.g., `cosine` takes one parameter), and standard data formats. These are fundamental lessons for broader reuse efforts.

14.3 Basic Issues in Any Reuse Program

Successfully implementing a reuse program requires addressing several key issues:

  • Component creation (identifying and designing for reusability).
  • Component indexing and storing (for efficient search, e.g., in RDBMS/ODBMS).
  • Component searching (matching requirements with available components).
  • Component understanding (clear documentation and well-defined functionality).
  • Component adaptation (modifying components to fit new contexts, which can be risky).
  • Repository maintenance (adding new, tracking faulty, retiring obsolete components).

14.4 A Reuse Approach

A promising approach is to introduce a "building block" methodology. Reusable components are identified after each project, enhanced for reusability, and cataloged into a component library.

14.4.1 Domain Analysis

The aim of domain analysis is to identify reusable components relevant to a specific problem domain (a technically related set of applications, e.g., accounting software, banking software). A reuse domain shows coherence through shared concepts, techniques, and terminologies.

Example: In an airline reservation system, reusable objects could be seats, flights, airports, crew; reusable operations could be scheduling a flight, reserving a seat, assigning crew. Domain analysis helps generalize application domains.

**Domain Engineering** is the actual construction of reusable components for a domain, often involving a community of developers discussing common solutions.

Evolution of a Reuse Domain (Stages):
Stage 1: No clear/consistent notations; no reusable components. All software from scratch.
Stage 2: Experience from similar projects; only knowledge reuse exists.
Stage 3: Domain concepts stabilized, notations standardized. Standard solutions to standard problems available. Ripe for both knowledge and component reuse.
Stage 4: Domain fully explored. Software development largely automated using domain-specific languages (DSLs) or application generators.

14.4.2 Component Classification

Proper classification is crucial for effective indexing and storage. Components can be described by natural language, logic schema, timing information, etc. Higher-level descriptions often have more ambiguity.

Prieto-Diaz's Classification Scheme:

A faceted classification where each component is described by a number of characteristics or facets, such as:

  • Actions they embody
  • Objects they manipulate
  • Data structures used
  • Systems they are part of

This requires enumerating an n-tuple that best fits the component, forcing a predefined hierarchy and making it easier to classify and search than purely enumerative schemes. Cross-referencing can handle complex relationships.

14.4.3 Searching

For large repositories, efficient searching is key. A popular method involves a web interface to the repository, using approximate automated search (keywords) and then browsing related items. This allows users to start with a broad search and progressively refine it. Remember that items to be searched can be components, designs, models, requirements, or even knowledge.

14.4.4 Repository Maintenance

This involves entering new items, retiring obsolete items, and modifying search attributes to improve search effectiveness. It's an ongoing process critical to the health of the reuse program.

14.4.5 Reuse without Modifications & Application Generators

The ideal is "plug-and-play" reuse where standard solutions emerge that require no modification. This is more useful than traditional program libraries and can be supported by compilers through linkage to run-time support routines (often called **application generators**).

Application generators translate specifications (which can be textual using a 4GL or visual using standard symbols) into application programs. They excel at parameterizing routines (defining what's variant vs. invariant).

Advantages of Application Generators:
  • Express variant information appropriately.
  • Result in fewer errors.
  • Easier to maintain programs.
  • Abstract away substantial development effort for implementation details.
Handicaps:
  • May not support all necessary new concepts/features (requiring an "escape mechanism" to a 3GL).
  • More successful for data processing and UI development; less so for applications with close hardware interaction or real-time systems.

14.5 Reuse at Organization Level

Reusability should be a standard part of all software development activities (design, implementation, testing, etc.), fostering a steady flow of reusable components. However, extracting reusable components from ongoing projects can be difficult for original developers.

Steps to Achieve Organization-Level Reuse:
  1. Assessing a product's potential for reuse: Often via questionnaires circulated among developers. Questions might include:
    • Would the component's functionality be required for future systems?
    • How common is its function within its domain?
    • Would there be duplication if this component isn't taken up?
    • Is it hardware dependent? Is the design optimized enough?
    • If non-reusable, can it be decomposed into reusable parts?
    • Can we parameterize a non-reusable component to make it reusable?
  2. Refining a product for greater reusability: Must be relatively easy to adapt.
    • Name generalization: General names, not application-specific.
    • Operation generalization: Add operations to make it more general; remove application-specific operations.
    • Exception generalization: Handle a broader range of exceptions.
  3. Handling portability problems: Programs often make assumptions about the underlying machine or OS. These machine dependencies (e.g., data representation, OS calls, function libraries) must be abstracted or localized.
14.5.1 Current State of Reuse (Inhibiting Factors)

Despite benefits, many organizations struggle with effective reuse programs. Non-technical factors are often the culprits:

  • **Need for commitment from top management.**
  • **Adequate documentation to support reuse.** (Crucial for understanding and trusting components).
  • **Incentives to reward those who contribute reusable components AND those who reuse existing components.**
  • **Providing access to and information about the reuse repository.** (Fear of competitors accessing proprietary components can be a barrier if not managed).
  • Restricting reuse to highly mature components can stifle creative potential. Making a product available before thorough assessment can be counterproductive ("Negative experiences tend to make one distrustful of the entire reuse framework").
