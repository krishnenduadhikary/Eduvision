Topic 10

User Interface (UI) Design

The user interface is the critical portion of a software product responsible for all interactions with the user. In an era of highly interactive software, users often judge a product by its UI. A well-designed UI enhances usability, reduces errors, and increases user satisfaction, while a poor UI can lead to frustration and inefficiency.

Development of a good UI can consume a significant portion (up to 50%) of the total system development effort. Therefore, a systematic approach to UI design is essential.

9.1 Characteristics of a Good User Interface

A good user interface is more than just aesthetically pleasing; it possesses several key characteristics that contribute to a positive user experience:

  • Speed of Learning: Easy to learn, avoiding complex syntax. Uses metaphors (e.g., shopping cart, cut/paste) and intuitive command names. Consistency and component-based design also aid learning.
  • Speed of Use: Efficient interaction, minimizing time and effort for tasks. Frequently used commands should be easily accessible.
  • Speed of Recall: Easy for intermittent users to remember command procedures. Metaphors and intuitive names help.
  • Error Prevention & Handling: Minimizes scope for errors. Prevents wrong value entry. Provides clear, polite, and specific error messages.
  • Attractiveness: Visually appealing and engaging. Graphics-based UIs often have an edge here.
  • Consistency: Uniform command names, procedures, and behavior across the interface helps users generalize knowledge.
  • Feedback: Provides timely system responses to user actions, especially for long-running processes.
  • Support for Multiple Skill Levels: Caters to both novice (ease of learning) and expert users (efficiency, shortcuts, macros).
  • Error Recovery (Undo Facility): Allows users to easily reverse actions, reducing anxiety and encouraging exploration.
  • User Guidance and Online Help: Provides assistance when users are stuck or want to learn more.

9.2 Basic Concepts in UI Design

9.2.1 User Guidance and Online Help

Effective help systems are crucial for user support.

  • Online Help System: Should be context-dependent (tailored to what the user is doing) and adapt to the user's experience level. It should leverage graphics/animation and not just be a copy of manual text.
  • Guidance Messages: Prompt users about next actions, system status, and progress of commands. Should be configurable for different user levels.
  • Error Messages: Must be polite, specific, and avoid ambiguity (e.g., "Invalid input" is poor). They should clearly state the problem and ideally suggest a solution.

9.3 Types of User Interfaces

User interfaces can be broadly classified into three categories, though modern applications often use a combination.

9.3.1 Command Language-Based Interface (CLI)

Users interact by typing commands using a specific language syntax. Was prevalent before graphical monitors. Still favored by many technical users.

  • Advantages: Fast interaction for experts, efficient for complex commands (composition), implementable on cheap terminals, easier to develop (using tools like lex/yacc). Uses fewer computer resources than GUI.
  • Disadvantages: Difficult to learn, requires memorization, prone to typing errors, keyboard-only interaction, unsuitable for casual users.
  • Design Issues: Meaningful mnemonics, user-defined aliases, command composition facility.

9.3.2 Menu-Based Interfaces

Users select commands from lists (menus), reducing the need to remember exact syntax. Interaction is often via a pointing device.

  • Advantages: No need to remember command syntax (recognition vs. recall), minimal typing. Good for occasional users.
  • Disadvantages: Slower for experts, command composition is difficult or impossible, can be cumbersome if the number of choices is large.
Structuring Menu Items:
  • Scrolling Menu: Used when all choices don't fit on screen. Commands should be highly correlated for easy location.
  • Walking Menu: Selecting a menu item causes a further menu (sub-menu) to appear, often adjacent. Can become a hierarchical tree.
  • Hierarchical Menu: Menu items organized in a tree. Can manage many choices but users might lose track if the tree is deep.

9.3.3 Direct Manipulation Interfaces (Iconic Interfaces)

Present the interface as visual models of objects. Users issue commands by performing actions directly on these visual representations (e.g., dragging a file icon to a trash icon).

  • Advantages: Icons easily recognized, often language-independent.
  • Disadvantages: Can be slow for experienced users, difficult for complex commands (e.g., deleting all files matching a pattern).

Graphical User Interface (GUI) - General Concepts

GUIs provide graphical means for users to interact with systems, combining hardware and software. While often more resource-consuming than CLIs, they allow for complex, efficient, and accurate interactions.

Common GUI Elements:
  • Window: A primary area for displaying application content. Can be moved, resized, minimized. May contain child windows.
  • Tabs: Allow multiple documents or views within a single window (Tabbed Document Interface).
  • Menu: Standard list of commands, usually at the top, can be hidden/shown.
  • Icon: Small picture representing an application or file. Double-clicking often opens it.
  • Cursor: Interacting device (mouse, touch pointer) representation on screen.
Application Specific GUI Components:
  • Application Window: Main window using OS-supplied constructs or custom elements.
  • Dialogue Box: Child window for messages or user input (e.g., confirmation for deletion).
  • Text-Box: Area for user to type text-based data.
  • Buttons: Imitate real-life buttons for submitting inputs.
Other Impressive GUI Components:
  • Radio Buttons: Select one option from a mutually exclusive set.
  • Check Boxes: Select one or more options.
  • List Box: Select one or more items from a displayed list.
  • Sliders: Adjust a value along a continuous range.
  • Combo Box: Button-like, expands to show a list of choices. Useful for space saving or run-time computed menus.
  • Data Grid: Displays data in a tabular (rows and columns) format.
  • Drop-down List: Similar to a combo box, presents a list of options when clicked.

9.4 Component-Based GUI Development

Modern GUI development relies heavily on pre-defined components (widgets). This approach became popular with the advent of graphical terminals and iconic interfaces in the 1980s (e.g., Apple Macintosh).

9.4.1 Window System

Most GUIs are built using a window system that manages displays through windows.

  • Window: A rectangular screen area, a "virtual screen" for independent activities. Divided into a client area (for application display) and a non-client area (borders, scroll bars, managed by the window manager).
  • Window Management System (WMS): Manages multiple windows, screen resources, and basic window behaviors (create, move, resize, iconify). Consists of a Window Manager and a Window System.

The **Window Manager** is the component the end-user interacts with for window operations. The **Window System** determines how windows look and behave, providing services to application programmers. **Widgets** are standard UI components (abstractions of window objects) used for development.

Component-based (widget-based) development helps users learn interfaces faster due to consistency across applications and significantly reduces the application programmer's work, making them more of an "interface integrator."

9.4.3 An Overview of X-Window/Motif

The X-Window system is notable for allowing development of network-independent GUIs. Applications developed with X-Window are device-independent and can run on one computer while displaying on another across a network.

9.4.4 X-Architecture

  • X-server: Runs on hardware, manages display and input devices. Performs low-level graphics, window, and input functions. Controls access to bit-mapped display.
  • X-protocol: Defines request format between client applications and display servers over the network. Independent of hardware, OS, and programming language.
  • X-library (Xlib): C library providing routines for applications to make requests (via X-protocol) to the X-server. Low-level primitives for displaying windows, drawing graphics, handling events.
  • Xtoolkit (Xt): Higher-level library built on Xlib. Consists of:
    • Intrinsics: Library routines to combine widgets into a UI.
    • Widgets: Pre-defined UI components (scroll bars, buttons). X/Motif is a popular widget set. DECWindows used X-Window for its look and feel.

9.4.5 Visual Programming

A drag-and-drop style of UI development. Visual objects (icons) representing GUI components are provided by the programming environment. The developer builds the UI by dragging and placing these components. Examples: Visual Basic, Visual C++.

Size Measurement of Component-Based GUI:

Lines of Code (LOC) is not suitable. **Widget Points (WP)** (total number of widgets) is a better metric, representing intricacy and being relatively environment-independent.

9.5 A User Interface Design Methodology

While no strict step-by-step methodology exists, user-centered design principles guide the process. This is distinct from design *by* users; developers leverage user knowledge but apply design expertise.

Implications of Human Cognition:
  • Limited Short-Term Memory: Humans remember ~7 unrelated items. UI should display necessary info, not require recall across screens.
  • Frequent Task Closure: Break complex tasks into subtasks with clear completion feedback for user satisfaction.
  • Recognition Rather Than Recall: Presenting options is better than requiring users to remember commands.
GUI Design Activities:
  1. GUI Requirement Gathering: Collect functional and non-functional UI requirements.
  2. User Analysis: Understand the target audience (technical savvy, experience) to tailor complexity.
  3. GUI Task Analysis: Analyze tasks to be done (what, not how). Represent hierarchically.
  4. GUI Design and Implementation: Design the UI based on requirements, tasks, user environment. Implement with working or dummy software.
  5. GUI Testing: In-house inspection, user involvement, beta releases. Test usability, compatibility, user acceptance.

GUI implementation tools (e.g., FLUID, AppInventor, Visual Studio) provide controls and allow code customization.

User Interface Golden Rules (Shneiderman & Plaisant)

These widely recognized principles provide a strong foundation for creating effective user interfaces:

Strive for consistency.

Consistent actions, terminology, and commands throughout.

Enable frequent users to use shortcuts.

Abbreviations, function keys, macros for expert users.

Offer informative feedback.

System response for every user action; modest for minor, substantial for major.

Design dialogs to yield closure.

Organize actions into groups with clear beginnings, middles, and ends for a sense of accomplishment.

Offer simple error handling.

Prevent serious errors; provide simple, comprehensible error detection and handling.

Permit easy reversal of actions.

Relieves anxiety and encourages exploration; allows undoing of single actions or groups.

Support internal locus of control.

Make users feel in charge, initiating actions rather than just responding.

Reduce short-term memory load.

Keep displays simple, consolidate multiple pages, reduce window motion, allow training time.
