Topic 15

Software Maintenance, Re-engineering & CASE Tools

Once software is delivered, its lifecycle is far from over. Software maintenance involves all modifications and updates post-delivery. Re-engineering aims to improve existing software, while Computer-Aided Software Engineering (CASE) tools provide automated support throughout the software development and maintenance processes.

Software Maintenance

Software maintenance encompasses all activities performed after the software product has been delivered to the customer. These modifications are essential to keep the software useful and operational over its lifespan.

Reasons for Maintenance:
  • Market Conditions: Changes in policies, taxation, regulations (e.g., new bookkeeping rules).
  • Client Requirements: Requests for new features, enhancements, or modifications to existing functionalities.
  • Host Modifications: Changes in the target hardware, operating system, or platform requiring software adaptation.
  • Organization Changes: Business-level changes at the client's end (e.g., mergers, new business ventures, downsizing) that necessitate software adjustments.

Types of Maintenance

Corrective Maintenance

Modifications and updates to correct or fix problems (bugs) discovered by users or identified through error reports.

Adaptive Maintenance

Modifications and updates to keep the software product current and compatible with a changing technological and business environment (e.g., OS updates, new hardware).

Perfective Maintenance

Modifications and updates to keep the software usable over a long period. Includes adding new features, accommodating new user requirements, and refining the software to improve its reliability and performance.

Preventive Maintenance

Modifications and updates to prevent future problems. Aims to attend to issues that are not significant at the moment but may cause serious problems later (e.g., code restructuring for better maintainability).

Cost of Maintenance

The cost of maintenance is notoriously high, often reported to be as much as 67% of the entire software process cycle, exceeding 50% of all SDLC phases on average.

Real-world factors affecting Maintenance Cost:
  • Standard age of software (10-15 years). Older software on slow machines with less memory struggles with modern hardware enhancements.
  • Advancing technology makes maintaining old software costly.
  • Many maintenance engineers might be newer and use trial-and-error, potentially harming the original structure.
  • Undocumented changes can cause future conflicts.
Software-end factors affecting Maintenance Cost:
  • Structure of the Software Program.
  • Programming Language used.
  • Dependence on external environment.
  • Staff reliability and availability.

Maintenance Activities (IEEE Framework)

IEEE provides a framework for a sequential maintenance process, which can be used iteratively and customized.

  1. Identification & Tracing: Identifying the need for modification or maintenance, often generated by user requests or system logs.
  2. Analysis: Analyzing the impact of the modification on the system (including safety/security). If severe, alternatives are sought. Requirements for the modification are specified, and cost estimations are made.
  3. Design: Designing new or modified modules based on the new requirements. Test cases are created.
  4. Implementation: Coding the new modules according to the design. Unit testing is performed by programmers.
  5. System Testing: Integration testing between new and existing modules. The system as a whole is tested using regression testing procedures.
  6. Acceptance Testing: Users test the system. Any issues are addressed or noted for future iterations.
  7. Delivery: Deploying the modified system (via update package or fresh install). Final testing at the client end.
  8. Maintenance Management: Essential configuration management, version control, and patch management.

Software Re-engineering

The process of software development undertaken to improve the maintainability of an existing software system, often by examining, altering, and reconstituting it in a new form. This is crucial when software needs to be updated to current market standards without altering its core functionality.

Objectives of Re-engineering:
  • Provide a cost-effective option for system evolution.
  • Improve maintainability and understandability.
  • Address issues related to obsolete technology or platforms.
Re-engineering Process Steps:
  1. Inventory Analysis: Decide what to re-engineer (whole software or parts).
  2. Document Reconstruction: Recreate or update documentation if missing or outdated.
  3. Reverse Engineering: Analyze the existing system to understand its components, interrelationships, and extract design/specification information. This is like a reverse SDLC, moving from code to higher abstraction levels.
  4. Code Re-construction/Restructuring: Inspect and restructure the source code using tools to remove anomalies and improve clarity, without changing its external behavior.
  5. Data Re-construction/Restructuring: Analyze and redefine data objects, attributes, and structures to make them more understandable and efficient.
  6. Forward Engineering (Renovation): Use the recovered design and specification information to modify or rebuild the system to achieve the desired improvements.
Need for Software Re-engineering:
  • Improving software product quality.
  • Cost-effective alternative to complete redevelopment.
  • Minimizing risk by building upon an existing, understood system.
  • Enabling software reusability by adding new features.
  • Enhancing software maintainability and operational life.
  • Adapting to new hardware, software platforms, or programming languages.
  • Rediscovering embedded business rules.
  • Better use of existing staff expertise.
Advantages:
  • Reduced Risk
  • Reduced Cost (vs. new development)
  • Revelation of Business Rules
  • Better use of Existing Staff
  • Increased performance, time-saving, new features possible, cost-reducing, longer operational life, clear documentation.
Disadvantages / Cost Factors:
  • Practical limits to the extent of re-engineering.
  • Quality of software to be re-engineered.
  • Tool support availability.
  • Extent of required data conversion.
  • Availability of expert staff.

Component Reusability

A component is a part of software program code that executes an independent task and can be a small module or a sub-system. Reusability aims to use existing components to build new software, reducing development time and cost, and improving quality.

Examples: Login procedures, printing systems. Components ideally have high cohesion and low coupling.

Levels of Re-use:
  • Application level: Entire application used as a sub-system.
  • Component level: Sub-system of an application is used.
  • Modules level: Functional modules are re-used.

This forms the basis of **Component-Based Software Engineering (CBSE)**.

Reuse Process:
  1. Requirement Specification: Functional and non-functional requirements are specified, considering existing system capabilities or user input.
  2. Design: Standard SDLC step; requirements are defined in terms of software parlance. Basic system architecture is created.
  3. Specify Components: Designers segregate the system into smaller components/sub-systems based on the software design.
  4. Search Suitable Components: The software component repository is searched for matching components based on functionality and intended use.
  5. Incorporate Components: All matched components are packaged together to form the complete software.

Computer-Aided Software Engineering (CASE) Tools

CASE tools are software applications used to automate and support SDLC activities, aiding project managers, analysts, and engineers in developing and maintaining software systems. They accelerate development, help uncover flaws early, and improve overall quality and consistency.

Key Components / Scope:
  • Central Repository: A central storage for all project information (specifications, requirements, reports, diagrams, data dictionary), ensuring common, integrated, and consistent data.
  • Upper CASE Tools: Used in early SDLC stages: planning, analysis, and design.
  • Lower CASE Tools: Used in later SDLC stages: implementation, testing, and maintenance.
  • Integrated CASE Tools: Span multiple or all SDLC stages, from requirements to documentation.
  • Diagram Tools: Represent system components, data, and control flow graphically (e.g., Flow Chart Maker).
  • Process Modeling Tools: Create software process models to develop or modify processes (e.g., EPF Composer).
Categories of CASE Tools & Examples:
Analysis Tools

Gather requirements, check for inconsistency, inaccuracy, data redundancies. Ex: Accept 360, Accompa, CaseComplete, Visible Analyst.

Design Tools

Design block structure, module detailing, interconnections. Ex: Animated Software Design.

Project Management Tools

Project planning, cost/effort estimation, scheduling, resource planning, real-time info sharing. Ex: Creative Pro Office, Trac Project, Basecamp.

Documentation Tools

Generate documents for technical users (manuals) and end-users (user guides). Ex: Doxygen, DrExplain, Adobe RoboHelp.

Configuration Management Tools

Version/revision management, baseline configuration, change control. Ex: Fossil, Git, AccuREV.

Change Control Tools

Part of CM; manage changes after baseline, automate tracking, file/code management, enforce change policy.

Programming Tools

IDEs, module libraries, simulation tools for building and testing. Ex: Cscope, Eclipse.

Prototyping Tools

Create simulated versions, initial look & feel, graphical libraries, hardware-independent UI design, rapid prototyping. Ex: Serena prototype composer, Mockup Builder.

Web Development Tools

Design web pages, forms, scripts, graphics, live preview. Ex: Fontello, Adobe Edge Inspect, Foundation 3, Brackets.

Quality Assurance Tools

Monitor engineering process, ensure quality conformance. Include CM, change control, testing tools. Ex: SoapTest, AppsWatch, JMeter.

Maintenance Tools

Automatic logging, error reporting, ticket generation, root cause analysis. Ex: Bugzilla, HP Quality Center.
