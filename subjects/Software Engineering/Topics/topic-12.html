Topic 11

Coding and Testing

The coding phase transforms the detailed design of a system into executable high-level language code. Following coding, rigorous testing is performed to evaluate the software against requirements, identify defects, and ensure a quality product. This involves both validating that the software meets user needs and verifying that it's built correctly according to specifications.

Software Validation: Process of examining if the software satisfies user requirements. Answers: "Are we developing the product which attempts all that user needs?" (Emphasizes user requirements).
Software Verification: Process of confirming if the software meets business requirements and adheres to design specifications and methodologies. Answers: "Are we developing this product by firmly following all design specifications?" (Concentrates on design/system specs).
Key Terms in Testing:
  • Error: A coding mistake made by a developer; a discrepancy between actual and desired output.
  • Fault (Bug): The result of an error, which can cause the system to behave incorrectly or fail.
  • Failure: The inability of the system to perform its desired task, occurring when a fault exists.

10.1 Coding

The input to this phase is the design document, including module structure and specifications (data structures, algorithms). The primary objective is to translate this design into high-level language code and perform initial unit tests on this code.

10.1.1 Coding Standards and Guidelines

Good software development organizations often establish coding standards and guidelines to ensure:

  • Uniform appearance and readability of code written by different engineers.
  • Improved code understandability and maintainability.
  • Encouragement of good programming practices.
Representative Coding Standards:
  • **Module Headers:** Include module name, creation date, author, modification history, synopsis, functions supported (with parameters), global variables accessed/modified.
  • **Naming Conventions:** Consistent rules for global variables (e.g., starting with uppercase), local variables (e.g., all lowercase), and constants (e.g., all uppercase).
  • **Error Return Conventions & Exception Handling:** Standardized ways for functions to report errors (e.g., consistently return 0 or 1) and handle common exceptions.
  • Rules for data types, global variable usage, code indentation, comments.
Representative Coding Guidelines:
  • Avoid overly clever or difficult-to-understand code. Clarity is paramount.
  • Avoid obscure side effects (e.g., unexpected modification of global variables or reference parameters).
  • Do not use an identifier for multiple, unrelated purposes. Give descriptive names.
  • Ensure code is well-documented (e.g., at least one comment line per three source lines as a rule of thumb).
  • Limit function length (e.g., not exceeding 100 source lines, though this varies) to maintain understandability.
  • Do not use `goto` statements, as they make programs unstructured and hard to follow.

10.2 Code Review

Performed after a module is compiled and syntax errors are eliminated. Code reviews are cost-effective for reducing coding errors and producing high-quality code.

10.2.1 Code Walk-Throughs

An informal code analysis technique. Team members read and manually simulate the execution of code with test cases before a meeting. The main objective is to discover algorithmic and logical errors. Discussion focuses on error discovery, not fixing. Managers typically don't attend to foster cooperation.

10.2.2 Code Inspection

A more formal technique aimed at discovering common types of errors due to oversight or improper programming, and checking adherence to coding standards. Uses checklists of common errors.

Classical Programming Errors Checked During Inspection:
  • Use of uninitialized variables.
  • Jumps into loops.
  • Non-terminating loops.
  • Incompatible assignments.
  • Array indices out of bounds.
  • Improper storage allocation/deallocation.
  • Mismatches between actual and formal parameters.
  • Incorrect logical operators or precedence.
  • Improper modification of loop variables.
  • Comparison of equality of floating-point values.

Detecting errors during code reviews or unit testing is much less costly to fix than if found during later system testing phases.

Software Testing Fundamentals

Manual Vs Automated Testing

  • Manual Testing: Performed by software testers without automated tools. Involves preparing test cases, executing tests, and reporting results. Time and resource-consuming.
  • Automated Testing: Uses tools to perform tests. Overcomes limitations of manual testing for tasks like load testing, stress testing, and regression testing.

Testing Approaches

  • Functionality Testing (Black-Box Testing): Tests functionality without considering internal implementation.
  • Implementation Testing (White-Box Testing): Tests functionality and analyzes the internal implementation (code structure, paths).

Exhaustive Testing (testing every possible input/output value) is ideal but practically impossible for most non-trivial software.

Levels of Software Testing

Testing is conducted at different stages of the software development lifecycle.

  1. Unit Testing: Testing individual modules or components of the source code in isolation, typically by developers in their environment. AKA Module Testing or Component Testing.
  2. Integration Testing: Testing the connectivity and data transfer between two or more unit-tested modules. Approaches include:
    • Top-Down: High-level modules first, using **Stubs** (dummy modules called by the module under test).
    • Bottom-Up: Low-level modules first, using **Drivers** (dummy modules that call the module to be tested).
    • Sandwich: Combination of Top-Down and Bottom-Up.
    • Big Bang: Integrate all modules at once (can make defect isolation difficult).
  3. System Testing (End-to-End Testing): Testing the fully integrated application as a whole. A black-box activity verifying that the software works in all intended target systems and meets requirements. Includes functionality, performance, security, and portability testing.
  4. Acceptance Testing: Performed to obtain customer sign-off before delivery.
    • Alpha Testing: Conducted at the developer's site, often by internal employees (testers/developers) or sometimes by clients in a controlled environment. Simulates real user tasks.
    • Beta Testing: Performed by a limited number of actual end-users in their real environment before the product's official release. Provides valuable feedback on product quality and usability.
    • Gamma Testing: (Less common) Testing done right before release, often skipping in-depth QA to focus on final checks.

Testing Techniques

Black-Box Testing (Behavioral/Specification-Based Testing)

Tests the functionality of a software/application without knowing its internal structure, design, or implementation details. The tester focuses on inputs and expected outputs.

Black-Box Testing Techniques:
Equivalence Class Partitioning (ECP)

Input values are divided into groups or "equivalence classes" based on expected similar outcomes. If one value from a class passes, all values in that class are assumed to pass. Reduces the number of test cases. Example: For an age field accepting 18-60, classes are <=17 (invalid), 18-60 (valid),>=61 (invalid).

Boundary Value Analysis (BVA)

Focuses on testing values at the boundaries (limits) of input domains, as errors often occur at these edges. Test cases include min, min-1, min+1, nominal, max-1, max, max+1. Example: For age 18-56, boundaries are 18 and 56. Test cases: 17, 18, 19, (some nominal), 55, 56, 57.

Decision Table Testing / Cause-Effect Graphing

Used when output depends on logical combinations of input conditions. A decision table lists conditions and the actions taken for each combination. Cause (input) - Effect (output) graphing helps identify these combinations systematically.

State Transition Testing

Tests the system's behavior as it transitions between different states based on input conditions or events. A state transition diagram helps visualize these changes. Effective for simpler applications.

Pair-wise Testing

Tests all possible pairs of input parameter values to reduce the number of test cases when dealing with multiple parameters, assuming most faults are caused by interactions between pairs of parameters.

Error Guessing

An experience-based technique where testers use their intuition and experience to guess error-prone areas (e.g., division by zero, null inputs, file upload limits).

Graph-Based Testing Methods

Identifies objects/components of an application, creates a graph representing their relationships, and designs test cases to cover these relationships.

Comparison Testing

Compares the output of the software under test with different independent versions of the same software or similar products.

White-Box Testing (Structural/Glass Box/Clear Box Testing)

Tests the internal structure, design, and implementation of the software. The tester has knowledge of the source code and focuses on testing paths, branches, statements, and conditions within the code.

White-Box Testing Techniques:
Control-Flow Testing

Aims to set up test cases that cover all statements and branch conditions. Involves creating a **Control Flow Graph (CFG)**, which describes the sequence of execution of program instructions.

Path Coverage: Design test cases such that all linearly independent paths in the program's CFG are executed at least once.

Data-Flow Testing

Focuses on covering all data variables included in the program. Tests where variables are declared, defined, used, and potentially changed or destroyed (definition-use pairs).

Cyclomatic Complexity (McCabe's Complexity)

A quantitative measure of the logical complexity of a program, based on its control flow graph. It indicates the number of linearly independent paths.

Formula: V(G) = e - n + 2P (where P is usually 1 for a single connected program).
Simpler: V(G) = Number of Edges (e) - Number of Nodes (n) + 2.

Debugging

Debugging is the process of finding and resolving defects or bugs in the software that cause it to behave incorrectly. It's a consequence of successful testing (i.e., tests that find failures). Debugging involves identifying the root cause of the error and then correcting the code.

Program Analysis Tools (Software Metrics)

Software metrics provide quantitative measures of various attributes of software or the software process. They can help in understanding complexity, estimating effort, and tracking quality.

Halstead's Complexity Measures

Introduced by Maurice Halstead in 1977, these metrics are computed directly from operators and operands in the source code (static analysis).

Halstead's Basic Parameters
Parameter Meaning
n1 Number of unique operators
n2 Number of unique operands
N1 Total number of occurrences of operators
N2 Total number of occurrences of operands
Halstead's Derived Metrics
Metric Meaning Mathematical Representation
n (Vocabulary) n1 + n2 Program Vocabulary
N (Size) N1 + N2 Program Length
V (Volume) N * log2(n) Program Volume (bits to represent)
D (Difficulty) (n1/2) * (N2/n2) Program Difficulty (to write/understand)
E (Effort) D * V Implementation Effort
B (Errors) V / 3000 (approx.) Estimated Number of Bugs
T (Testing Time) E / S (where S=18 seconds/Stroud number) Time to implement or understand

Function Point (FP) Analysis

A method to measure software size based on the functionality provided by the system from a user's perspective. FP counts are based on five parameters, each categorized as Simple, Average, or Complex.

Function Point Parameters & Weightages:
Parameter Simple Average Complex
External Inputs (EI) 3 4 6
External Outputs (EO) 4 5 7
External Inquiries (EQ) 3 4 6
Logical Internal Files (ILF) 7 10 15
External Interface Files (EIF) 5 7 10

The sum of weighted parameters gives Raw Function Points (Raw FP). This is then adjusted by a Complexity Adjustment Factor (CAF), derived from rating 14 General System Characteristics (GSCs) like data communications, performance, reusability, etc., on a scale of 0 (no influence) to 5 (strong influence).
CAF = 0.65 + (0.01 * N), where N is the sum of GSC ratings.
Delivered Function Points (FP) = CAF * Raw FP.
FP can then be used for metrics like Cost/FP, Errors/FP, FP/person-month.
