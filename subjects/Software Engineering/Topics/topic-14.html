Topic 13

Computer-Aided Software Engineering (CASE)

CASE tools help make the software development process more efficient. As software becomes an increasingly costly component due to rising manpower involvement in development and maintenance, CASE tools promise reductions in these costs and aid in developing better quality products more effectively.

12.1 CASE and Its Scope

A **CASE tool** is a generic term for any form of automated support for software engineering. More restrictively, it can mean any tool used to automate some activity associated with software development. Many CASE tools are available, some assisting in phase-related tasks (specification, structured analysis, design, coding, testing) and others in non-phase activities (project management, configuration management).

Primary Objectives of Deploying CASE Tools:
  • To increase productivity.
  • To produce better quality software at lower cost.

12.2 CASE Environment

While individual CASE tools are useful, their true power is realized when integrated into a **CASE Environment** through a common framework and a **central repository**. If tools are not integrated, data generated by one might require format conversions for use by another, leading to extra effort and potential data maintenance issues in proprietary formats.

A CASE environment facilitates the automation of step-by-step methodologies for software development. It shares common information (often via a data dictionary in the central repository) among all its composite and elementary data.

This contrasts with a **Programming Environment** (e.g., Turbo C, Visual Basic), which is an integrated collection of tools supporting only the coding phase (text editor, compiler, debugger).

12.2.1 Benefits of CASE

  • **Cost Savings:** Significant effort reduction (30-40%) across developmental phases.
  • **Improved Quality:** Facilitates iteration through development phases, reducing chances of human error.
  • **High-Quality, Consistent Documents:** Central repository maintains data, reducing redundancy and inconsistencies.
  • **Reduced Drudgery:** Automates laborious tasks (e.g., DFD balancing).
  • **Maintenance Cost Savings:** Tremendous value in visibility and consistency checks, systematic information capture.
  • **Improved Working Style:** Encourages a conscious, structured, and orderly approach within a company.

12.3 CASE Support in Software Life Cycle

CASE tools should support a development methodology, enforce the same, and provide consistency checking between different phases.

12.3.1 Prototyping Support

Prototyping helps understand requirements, demonstrate concepts, and market ideas. Prototyping CASE tools should:

  • Define user interaction (GUI creation via graphics editor for forms, menus, controls).
  • Define system control flow.
  • Store and retrieve data required by the system.
  • Incorporate some processing logic.
  • Integrate with the CASE environment's data dictionary.
  • Allow integration with external user-defined modules (e.g., written in C).
  • Enable definition and control of prototype execution sequence.
  • Support mock-up runs of the actual system (input/output management).

12.3.2 Structured Analysis and Design Support

  • Support for one or more structured analysis and design techniques and diagrams.
  • Effortless creation of complex diagrams, preferably hierarchically.
  • Easy navigation through different levels of diagrams.
  • Provide completeness and consistency checking across design and analysis levels (though full, rigorous checking can be computationally intensive).

12.3.3 Code Generation Support

Traceability from source code to design data is a key expectation. Pragmatic support includes:

  • Generation of module skeletons or templates in popular programming languages.
  • Automatic generation of record/structure/class definitions from the data dictionary.
  • Generation of database tables for RDBMS.
  • Code generation for user interfaces from prototype definitions (e.g., for X-Windows, MS Window-based applications).
  • Automatic generation of copyright messages, module author name, and creation date in a selectable format.

12.3.4 Test Case Generator Support

  • Should support both design and requirement testing.
  • Should generate test set reports in an ASCII format (or other standard format) that can be directly imported into test plan documents.

12.4 Other Characteristics of CASE Tools

These features enhance the effectivity and usefulness of CASE tools, though not central to their core functionality.

12.4.1 Hardware & Environmental Requirements

Focus on optimal tool configuration for existing hardware. Heterogeneous network support (client-server, distributed data access, multi-user, data dictionary consistency, backup/restore, multi-windowing environment) is increasingly important.

12.4.2 Documentation Support

Organized graphical and textual document generation from the central repository. Integration with DTP packages and export in standard formats (e.g., PostScript).

12.4.3 Project Management

Support for collecting, storing, and analyzing project progress data (schedules, task durations, review results).

12.4.4 External Interface

Allow information exchange for reusability (ASCII format, open architecture). Programming interface to data dictionary for custom utilities.

12.4.5 Reverse Engineering Support

Generate structure charts and data dictionaries from source code. Tools for re-engineering information systems (e.g., converting file structures).

12.4.6 Data Dictionary Interface

Provide view/update access to data items and relations. Support for print, cross-referencing, impact analysis, screen reports, and a query language.

12.4.7 Tutorial and Help

Crucial for user adoption. Should cover all tool techniques and facilities, logically classified, and be supported by proper documentation.

12.5 Towards Second Generation CASE Tools

An important desired feature is direct support for any adapted methodology, allowing a CASE administrator to tailor the tool. Key features include:

  • Intelligent Diagramming Support: Aesthetically pleasing and automatic layout of diagrams.
  • Integration with Implementation Environment: Seamless flow between design and implementation tools.
  • Data Dictionary Standards Support: Allowing integration of tools from different vendors via a common standard.
  • Customization Support: Ability for users to define new types of objects, connections, and build special methodologies, ideally with a rule engine for consistency checks.

12.6 Architecture of a CASE Environment

A typical modern CASE environment comprises several key components:

  • User Interface: Provides a consistent framework for accessing different tools, reducing learning time.
  • Tool Set: The collection of individual CASE tools that perform specific tasks (e.g., diagramming, code generation, analysis).
  • Object Management System (OMS): Maps logical entities (specifications, designs, project plans, etc.) into the underlying storage management system (the repository). CASE tools often create many types of entities with few instances, differing from RDBMS which handle large volumes of simple records.
  • Repository (Central Repository): The underlying storage for all project information, including the data dictionary. It ensures data consistency and sharing across tools.
