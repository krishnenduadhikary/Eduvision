Topic 12

Software Reliability and Quality Management

Ensuring software reliability and maintaining high quality are paramount in software engineering. This topic explores concepts related to how trustworthy a software product is, how its quality is defined and measured, and the management systems and models used to achieve these goals.

11.1 Software Reliability

Reliability of a software product essentially denotes its **trustworthiness** or **dependability**. It can also be defined as the probability of a product working correctly over a given period of time. A product with many defects is unreliable. While reliability improves as defects are reduced, there's no simple relationship between observed errors and latent defects.

It has been experimentally observed that often 90% of execution time is spent in only 10% of the program instructions (the 'core'). Removing defects from rarely used parts might only yield a small (e.g., 3%) improvement in overall reliability. Reliability depends not just on latent errors but also on how frequently the corrected portion of code is executed (its execution profile).

Perceived reliability is observer-dependent; different users might invoke different functionalities, encountering errors differently. Thus, software reliability cannot be determined absolutely.

Difficulties in Measuring Reliability:
  • Fixing a bug depends on where the bug is located in the code.
  • Perceived reliability is highly observer-dependent.
  • Reliability keeps changing as errors are detected and fixed (unlike hardware reliability which degrades over time).

11.1.1 Hardware vs. Software Reliability

Hardware and software reliability behaviors differ significantly. Hardware failures are often due to component wear and tear or short-circuits. Software failures stem from design or coding errors. Fixing hardware might involve replacement; fixing software means code changes, which can inadvertently introduce new errors (reliability might decrease before increasing).

Hardware reliability study often concerns stability (inter-failure durations constant), while software reliability study aims at reliability growth (inter-failure durations increase as bugs are fixed).

11.1.2 Reliability Metrics

Quantitative measures are needed to express software reliability. Key metrics include:

  • Rate of Occurrence of Failure (ROCOF): Frequency of unexpected behavior (failures) observed over a specified time interval.
  • Mean Time To Failure (MTTF): Average time between two successive failures, observed over many failures. MTTF = Σ(ti+1 - ti) / (n-1). Considers run-time only.
  • Mean Time To Repair (MTTR): Average time taken to track the cause of a failure and fix it.
  • Mean Time Between Failures (MTBF): MTBF = MTTF + MTTR. Indicates expected time until next failure.
  • Probability of Failure on Demand (POFOD): Likelihood of system failing when a service request is made. Does not involve time measurements. E.g., POFOD of 0.001 means 1 out of 1000 requests might fail.
  • Availability: Measure of how likely the system is available for use over a period. Considers failures and repair time (downtime). Availability = MTTF / (MTTF + MTTR) * 100%.
Types of Failures:
  • Transient: Occur only for certain input values while invoking a function.
  • Permanent: Occur for all input values for a function.
  • Recoverable: System recovers with or without operator intervention.
  • Unrecoverable: System may need to be restarted.
  • Cosmetic: Minor irritations, do not lead to incorrect results (e.g., UI glitch).

11.1.3 Reliability Growth Modelling

A mathematical model of how software reliability improves as errors are detected and repaired. Used to predict when a certain reliability level might be attained or when to stop testing.

Jelinski and Moranda model

Simplest model; assumes reliability increases by a constant increment each time an error is detected and repaired. Unrealistic as it assumes all errors contribute equally to reliability growth.

Littlewood and Verrall's model

Allows for negative reliability growth (repair might introduce new errors). Models decreasing average improvement per repair (errors contribute differently, using Gamma distribution). Represents diminishing returns as testing continues.

More complex models exist but are beyond this scope.

11.2 Statistical Testing

A testing process whose objective is to determine the reliability of the product rather than discovering individual errors. This has an entirely different objective from conventional functional testing.

Operation Profile:

Defines how different categories of users might use the software. For example, in a Library Automation Software, some users might primarily add books, others query availability, and some manage member records. An operation profile assigns a probability value {Pi} to each input class {Ci}, representing the likelihood of an average user selecting their next input from that class. This profile helps determine the operational use pattern of the software.

Steps in Statistical Testing:
  1. Define the operation profile of the product based on user categories and their input patterns.
  2. Generate a set of test data corresponding to the determined operation profile.
  3. Apply the test cases to the software and record the operation profile (time between each failure).
  4. After a statistically significant number of failures have been observed, the reliability can be computed.

11.3 Software Quality

Traditionally, a quality product is defined by its **fitness for purpose** – it does exactly what users want, often interpreted as satisfying requirements in the SRS document. However, this is not wholly satisfactory for software. A functionally correct product with an unusable UI or unmaintainable code cannot be considered a quality product.

The modern view associates software quality with several **quality factors**:

Portability: Can be easily made to work in different operating systems, machines, or with other software.
Usability: Different categories of users (expert/novice) can easily invoke functions. Good learnability.
Reusability: Different modules can be easily reused to develop new products.
Correctness: All requirements specified in the SRS are correctly implemented.
Maintainability: Easy to fix bugs, adapt to new requirements, and make changes.
Efficiency: Performs tasks without wasting resources (time, memory).
Reliability: Performs intended functions consistently under stated conditions (covered earlier).
Integrity/Security: Protects software and data from unauthorized access or modification.
Testability: Can be easily and thoroughly tested.

11.4 Software Quality Management System (SQMS)

A SQMS is the principal methodology used by organizations to ensure their products have the desired quality. It consists of:

  • Managerial Structure and Individual Responsibilities: Quality is an organizational responsibility, often with a dedicated quality department. Top management support is crucial for the quality system to be taken seriously.
  • Quality System Activities:
    • Auditing of projects.
    • Review of the quality system itself.
    • Development of standards, procedures, and guidelines.
    • Production of reports for top management on the effectiveness of the quality system.

A good SQMS must be well-documented. Undocumented quality controls lead to ad hoc results and variations in product quality. International standards like ISO 9000 provide guidance on organizing a quality system.

11.6 SEI Capability Maturity Model (CMM)

Proposed by the Software Engineering Institute (SEI) at Carnegie Mellon University, CMM is a framework for assessing and improving software process maturity. It helps organizations understand their current process capabilities and provides a roadmap for improvement.

Uses of SEI CMM:
  • Capability Evaluation: Assesses the likely contractor performance if awarded a work (external use, e.g., by DoD for contractor selection).
  • Software Process Assessment: Used by an organization to improve its own process maturity (internal use).
The Five Maturity Levels:
Level 1: Initial

Characterized by ad hoc, often chaotic activities. Few processes are defined, and success depends heavily on individual effort and heroics. Project management practices are not followed rigorously.

Level 2: Repeatable

Basic project management processes are established to track cost, schedule, and functionality. Process discipline exists to repeat earlier successes on similar projects. A process exists only when a company produces a family of products.

Level 3: Defined

Processes for both management and development activities are documented, standardized, and integrated into an organization-wide software process. Common understanding of activities, roles, and responsibilities. ISO 9000 aims at achieving this level.

Level 4: Managed

Focus is on software metrics. Detailed measures of the software process and product quality are collected. Both software process and products are quantitatively understood and controlled.

Level 5: Optimizing

Continuous process improvement is enabled by quantitative feedback from the process and from piloting innovative ideas and technologies. Lessons learned are systematically applied. The organization identifies and institutionalizes best software engineering practices.

Key Process Areas (KPAs):

Each maturity level (except Level 1) is characterized by several KPAs that indicate areas an organization should focus on to improve its process to the next level.

CMM Level Focus Key Process Areas
Initial (1) Competent people (None specified - implicit)
Repeatable (2) Project management Requirements Management, Software Project Planning, Software Project Tracking & Oversight, Software Subcontract Management, Software Quality Assurance, Software Configuration Management.
Defined (3) Definition of processes Organization Process Focus, Organization Process Definition, Training Program, Integrated Software Management, Software Product Engineering, Intergroup Coordination, Peer Reviews.
Managed (4) Product and process quality Quantitative Process Management, Software Quality Management.
Optimizing (5) Continuous process improvement Defect Prevention, Technology Change Management, Process Change Management.

SEI CMM doesn't dictate *how* to perform software development but suggests *what* processes are important for effective practices. The process for individual use is different from that for a team.

11.7 Personal Software Process (PSP)

Developed by Watts Humphrey, PSP is a structured framework of process disciplines for individual software engineers. It helps engineers measure, improve, and manage their personal development skills and project work.

Key Aspects of PSP:
  • Time Measurement: Engineers track time spent on tasks (designing, coding, testing, etc.) using a stop-clock, avoiding stretched or interrupted time logging.
  • PSP Planning: Individuals plan their projects, estimating maximum, minimum, and average LOC; productivity (minutes/LOC); and development time. Data is recorded in a project plan summary.
  • Logs: Defect logs, time logs are meticulously maintained. Post-mortem analysis of log data helps improve future planning.
Levels of PSP:

PSP levels (PSP0, PSP1, PSP2, PSP2.1, PSP3) introduce increasingly sophisticated process disciplines, including size/time estimation, planning, design/code reviews, and quality management. Checklists are used for defect management based on past project data.

11.8 Six Sigma

A disciplined, data-driven approach and methodology for eliminating defects in any process – from manufacturing to transactional and from product to service. It aims for near perfection, statistically meaning a process must not produce more than 3-4 defects per million opportunities (DPMO).

The fundamental objective is a measurement-based strategy focusing on process improvement and variation reduction. This is accomplished through Six Sigma improvement projects using two sub-methodologies:

  • DMAIC: Define, Measure, Analyze, Improve, Control. An improvement system for existing processes falling below specification.
  • DMADV: Define, Measure, Analyze, Design, Verify. An improvement system used to develop new processes or products at Six Sigma quality levels. (Also known as DFSS - Design For Six Sigma).

Six Sigma processes are executed by certified professionals: Green Belts and Black Belts, overseen by Master Black Belts.
