Topic 9

Object-Oriented Software Development

Object-Oriented Design (OOD) is a paradigm that shifts the focus from functions to the natural objects (entities) within a problem domain. This chapter builds upon object-modeling concepts to explore a comprehensive object-oriented analysis and design methodology.

Grady Booch famously summarized the core difference: "Read the specification. If you want procedural code, find the verbs. If you want object-oriented programs, find the nouns."

Object-Oriented Analysis (OOA) develops an object model from requirements, which is then refined into a design model (OOD) suitable for implementation using Object-Oriented Programming (OOP).

8.1 Design Patterns

Design patterns are reusable, well-documented solutions to commonly occurring problems within a given context in software design. They act as guides for creating good designs, based on fundamental principles and experience. Knowing patterns helps in developing robust solutions more quickly.

A pattern typically has four important parts:
  • The **Problem** it addresses.
  • The **Context** in which the problem occurs.
  • The **Solution** (often described in terms of class interactions).
  • The **Context** within which the solution works (and sometimes, where it might not).

Expert Pattern

Problem: Which class should be responsible for certain information or tasks?

Solution: Assign responsibility to the **information expert** – the class that has the information necessary to fulfill the required responsibility. This aligns with the intuition that objects "do" things related to the information they possess.

Creator Pattern

Problem: Which class should be responsible for creating an instance of class C2?

Solution: Assign class C1 the responsibility to create an instance of C2 if one or more of these apply:

  • C1 aggregates objects of type C2.
  • C1 contains objects of type C2.
  • C1 closely uses objects of type C2.
  • C1 has the data that will be required to initialize objects of type C2 when they are created.

Controller Pattern

Problem: Who should be responsible for handling actor requests (system events)?

Solution: For each use case, there should typically be a separate controller object responsible for handling all actor requests pertaining to that use case. This controller maintains necessary information about the state of the use case and can identify out-of-sequence requests. It decouples boundary objects (UI) from entity objects (domain logic).

Facade Pattern

Problem: How should services be requested from a complex package or subsystem?

Solution: Create a Facade class that provides a single, simplified, common interface to the services of the package. A package is a cohesive set of classes with strong responsibilities (e.g., an RDBMS interface). The Facade hides the complexity of the subsystem.

Model-View Separation Pattern

Problem: How should non-GUI (model/domain) classes communicate with GUI (view) classes without direct coupling?

Solution: Model objects should not have direct knowledge of or be directly coupled to view objects. This results in a reusable model. Two common approaches:

  1. Polling (Pull from above): GUI objects are responsible for asking (pulling) relevant information from model objects when needed. Efficient for some applications but not for real-time visualizations or monitoring.
  2. Publish-Subscribe Pattern (Push from below): An event notification system is implemented. An Event Manager class tracks subscribers and event types. Publishers send messages to the Event Manager, which then notifies all registered subscribers (often via callbacks). Java's `EventListener` interface supports this.

Intermediary Pattern (Proxy)

Problem: How should client and server objects (software components, possibly across a network) interact?

Solution: Define a Proxy object on the client side that acts as a local stand-in for the remote server object. The Proxy hides network transmission details, determines server address, communicates requests, obtains responses, and passes them to the client. It can also augment or filter exchanged information and may have the same interface as the remote server, abstracting network complexities from the client.

A Generalized Object-Oriented Analysis and Design Process

This process begins by making an analysis model, which is then refined into a design model through iterations. It explains how to define system requirements and translate them into a logical solution that can be implemented.

8.2.1 Overview

The process typically starts with developing a **Use Case Model**. This model, along with the **SRS document**, informs the construction of a **Domain Model**. The Domain Model is then refined into a **Class Diagram** through iterations involving **Interaction Diagrams**. A **Glossary** of terms is continuously maintained to ensure clear communication and reduce misunderstandings. User interface prototypes often evolve in parallel.

8.2.2 Use Case Model Development

The essence is to capture user requirements by detailing all scenarios of user interactions. There's a strong correlation between the user manual's content and the use case model. Iterating on GUI design and use cases in parallel is an excellent approach, but use cases should remain independent of specific GUI elements (e.g., radio buttons), as UI can change more frequently than core functionality. State use cases from the actor's perspective using active voice verb phrases (e.g., "Admit patient," "Issue book").

Common Mistakes in Use Case Model Development:
  • Clutter: Too many use cases in the top-level diagram. Organize using packages.
  • Confusing Sub-steps: Mistaking sub-steps of a use case (e.g., "print receipt") for separate, top-level use cases.
  • Omitting Text Descriptions: Makes it hard to fully understand behavior and leads to implementation difficulties.
  • Missing Alternative Scenarios: Leads to incomplete functionality or late discovery of issues.

8.2.3 Domain Modelling (Conceptual Modelling)

A domain model represents the concepts or objects appearing in the problem domain and their relationships. A recommended strategy is to quickly create a rough conceptual model focusing on obvious concepts, then refine and extend it.

Classifying Objects in the Domain Model:
  • Boundary Objects: Objects with which actors interact (e.g., screens, menus, dialogs). Responsible for user interaction, input validation, output formatting, but typically not processing logic. A common heuristic is one boundary class per actor/use case pair.
  • Entity Objects: Hold information that needs to outlive use case execution (e.g., data tables, files like `Book`, `BookRegister`). Often "dumb servers" for storing and fetching data.
  • Controller Objects: Coordinate activities of boundary and entity objects to provide overall system behavior for a use case. They decouple boundary and entity objects, embodying much of the use case realization logic. Often, one controller object per use case, though complex use cases might need more, or trivial ones might omit it.

8.2.4 Identification of Entity Objects

This is a crucial step. Grammatical analysis of the problem description (processing narrative) is a common approach (originated by Booch):

  • Nouns are potential objects.
  • Verbs are potential methods.

Eliminate synonyms. Distinguish between objects in the solution space (needed for implementation) and problem space (descriptive). An imperative verb form (e.g., "move") represents an action, not an object.

Criteria for Legitimate Objects:
  • Retained Information: Object needs to store some private data.
  • Multiple Attributes & Operations: Useful objects typically have several attributes and support multiple methods.
  • Common Operations: Defined operations must apply to all instances. If not, consider subclasses.

Actor Surrogates: Classes needed to store information about an actor within the system (e.g., `LibraryMember` class for a library member actor).

While grammatical analysis is simple, high-quality abstractions often result from deeper insight into the problem.

Example 8.1: Tic-Tac-Toe Entity Object Identification

Problem description nouns: Tic-tac-toe, game, human player, computer, moves, square, mark, straight line, board, row, column, diagonal. After eliminating synonyms, problem-domain-only nouns (human player), and action-nouns (move), the primary entity object identified is often `Board` (or `Square` as part of Board).

8.2.5 Interaction Modelling

Aims to allocate responsibilities to classes and show detailed interactions over time for each use case. This is typically done using **Sequence Diagrams**. One sequence diagram per use case is often needed, potentially split for complex scenarios or alternative flows.

8.2.7 CRC (Class-Responsibility-Collaborator) Cards

A simple, effective technique for determining class responsibilities in a use case realization. Pioneered by Ward Cunningham and Kent Beck.

  • Index cards, one per class.
  • Lists class name, its responsibilities, and collaborator classes it needs to fulfill those responsibilities.
  • Developed in small group sessions with role-playing.
  • Cards are deliberately small to limit responsibilities per class.

Interaction diagrams can often be developed by "flipping through" CRC cards.

8.4 OOD Goodness Criteria

Arriving at a good object-oriented design involves subjective judgments. Therefore, accepted criteria are needed to judge design quality.

  • Coupling Guidelines: Minimize coupling between objects or groups of objects. Excessive coupling is detrimental to modularity and prevents reuse.
  • Cohesion Guidelines: Aim for high cohesion at three levels:
    1. Cohesiveness of individual methods (each method does one well-defined function).
    2. Cohesiveness of data and methods within a class (methods act on data for which the object is naturally responsible).
    3. Cohesiveness of an entire class hierarchy (promotes encapsulation).
  • Hierarchy and Factoring Guidelines: A base class should not have too many subclasses (approx. 7±2 is a heuristic). This aids understandability.
  • Keeping Message Protocols Simple: Complex message protocols (e.g., more than three parameters) can indicate excessive coupling.
  • Number of Methods per Class: Objects with too many methods (e.g., >7) are likely application-specific, complex, and less reusable.
  • Depth of Inheritance Tree: Very deep trees can make classes complex due to many inherited methods. Keep height manageable.
  • Number of Messages per Use Case: A single message triggering an excessive chain of messages across many objects complicates testing and debugging.
  • Response For a Class (RFC): The maximum number of different methods an instance of this class would call (counting a method only once if called multiple times). A class calling >7 different methods might be error-prone.
