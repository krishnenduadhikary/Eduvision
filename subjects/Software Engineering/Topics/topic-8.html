Topic 8

Data Flow Diagrams (DFD)

A Data Flow Diagram (DFD) is a traditional graphical tool used to represent the flow of information within a system. It clearly depicts how data enters and leaves the system, what processes transform the information, and where data is stored. DFDs are essential for showing the scope and boundaries of a system and serve as a communication tool between analysts and stakeholders.

Key Observations about DFDs:
  • All names (processes, data stores, data flows, entities) should be unique for easy reference.
  • DFDs are NOT flowcharts. Arrows in DFDs represent flowing data, not the order of events or control flow.
  • Suppress logical decisions (no diamond-shaped boxes). DFDs don't imply an order of events for decisions.
  • Start with a high-level view and defer minute details, error conditions, and error handling until later stages of analysis or design.

Symbols and Conventions

Standard symbols are used to construct DFDs, primarily derived from Yourdon/DeMarco or Gane & Sarson notations. The core components are:

  • Process: Represented by a circle (bubble) or a rounded rectangle. It shows a process that transforms data inputs into data outputs.
  • Data Flow: Represented by a curved or straight line with an arrowhead. It shows the direction of data movement into or out of a process or data store.
  • Data Store: Represented by two parallel lines or an open-ended rectangle. It's a repository for data items that can be used later or by other processes.
  • External Entity (Source/Sink): Represented by a rectangle. It's an external system, person, or organization that acts as a source of system inputs or a sink for system outputs. It lies outside the boundary of the system being modeled.

Levels in Data Flow Diagrams (DFD)

DFDs are partitioned into levels to represent increasing information flow and functional detail. This hierarchical decomposition helps in understanding complex systems by starting from a high-level abstraction and progressively refining it.

0-level DFD (Context Diagram)

Also known as the fundamental system model or context diagram, it represents the entire software requirement as a single system (one process bubble) with input and output data denoted by incoming and outgoing arrows from/to external entities. It shows the system's boundary and its interaction with the external world, concealing all internal details. Preserving the number of inputs and outputs between levels is essential (levelling by DeMarco).

2-level DFD (and beyond)

A 2-level DFD goes one process deeper into parts of a 1-level DFD. Any process from a higher-level DFD can be further decomposed into a more detailed lower-level DFD. This process of decomposition can be repeated for as many levels as necessary until the program at hand is well understood and each process represents a reasonably simple function.

Balancing and Decomposition

A crucial principle in creating leveled DFDs is **balancing**. This means that a child diagram (e.g., Level 1 DFD decomposing a Level 0 process) must maintain the same net inputs and outputs as its parent process. If a process 'P' in a higher-level DFD has inputs I1, I2 and output O1, then the lower-level DFD that decomposes 'P' must collectively show I1, I2 entering its processes and O1 exiting from its processes. No new external inputs or outputs should appear at lower levels that were not present at the parent level for that specific process decomposition.

Decomposition continues until processes are simple enough to be easily understood and implemented (often called "primitive processes").

Guidelines for Developing Data Flow Diagrams

  • All processes in a DFD must go to another process or a data store (not directly to an external entity without passing through a process if it's an internal transformation).
  • Each process must have at least one input data flow and at least one output data flow.
  • Data stored in a system (in a data store) must be processed (i.e., a process must put data into it, and a process must get data out of it, though not necessarily the same process or at the same time).
  • Each data store must have at least one data flow in and at least one data flow out (though these may occur at different times or in different DFDs representing different system interactions).
  • Data flows should be named to reflect the data they carry. Process names should be verb phrases. Data store and entity names should be nouns.

Logical DFD vs. Physical DFD

DFDs can be categorized based on the perspective they represent:

Feature Logical DFD Physical DFD
Focus Describes business events, data required, and what the business/system does. Focuses on "what". Shows how the system will be implemented (programs, modules, manual procedures, physical files). Focuses on "how".
Processes Represent business activities. Represent programs, program modules, manual procedures.
Data Stores Represent collections of data regardless of how they are stored (e.g., "Customer Information"). Represent physical files, databases, manual files (e.g., "Customer_Master_Table", "Order_Forms_File_Cabinet").
Controls Shows business controls. Shows controls for data validation, record access, process completion, system security.
