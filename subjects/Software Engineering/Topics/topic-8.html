Topic 8

Object Modelling Using UML

Object-oriented software development has become a dominant paradigm, offering powerful ways to model and build complex systems. This topic explores core object-oriented concepts and introduces the Unified Modeling Language (UML) as a standard tool for visualizing, specifying, constructing, and documenting object-oriented systems.

7.1 Overview of Object-Oriented Concepts

Before diving into UML, let's revisit some fundamental concepts central to the object-oriented approach. These concepts define how we think about and structure software in an OO way.

Core OO Principles:
  • Objects: Real-world or conceptual entities involved in the solution design (e.g., person, bank, customer). Each object has attributes (characteristics) and methods (behaviors).
  • Classes: A generalized description or blueprint of an object. An object is an instance of a class. Classes define attributes and methods for their objects.
  • Encapsulation (Information Hiding): Bundling attributes (data) and methods (operations on data) together within an object, restricting direct access from outside. This protects internal state and promotes modularity.
  • Inheritance: Allows similar classes to be organized hierarchically. Sub-classes (derived classes) can inherit, implement, and reuse attributes and methods from their super-classes (base classes). This promotes code reuse and a natural way to model "is-a" relationships.
  • Polymorphism ("Many Forms"): Allows methods with similar tasks but different implementations (e.g., varying arguments or operating on different data types) to share the same name. The correct method is invoked based on the context or object type, enabling a single interface for diverse behaviors.
  • Abstraction: Focusing on essential characteristics while ignoring irrelevant details. Objects and classes provide abstractions of real-world entities.
  • Messages & Methods: Objects communicate by sending messages to each other. A message is a request for an object to perform one of its operations (methods).
Object-Oriented Design Process (High-Level):
  1. A solution design is created from requirements or existing system analysis.
  2. Objects are identified and grouped into classes based on shared attributes and characteristics.
  3. Class hierarchies and relationships among classes are defined (e.g., inheritance, association).
  4. An application framework, defining the overall structure and flow, is established.

7.2 Unified Modeling Language (UML)

UML is a standardized graphical modeling language used to specify, visualize, construct, and document the artifacts of a software-intensive system. It provides a rich set of notations (rectangles, lines, ellipses, etc.) to create various diagrams that model different aspects of a system.

Important Note: UML is a modeling language, NOT a system design or development methodology. It is also not tied to any specific object-oriented methodology, unlike its predecessors. Its independence allows it to be used with various design approaches.

Origin and Influences:

In the late 1980s and early 1990s, numerous object-oriented design techniques and notations proliferated, leading to confusion when different teams used different notations. UML was developed to standardize these, drawing heavily from established methodologies, primarily:

  • OMT (Object Modeling Technique) by Rumbaugh et al.
  • Booch's Methodology by Grady Booch.
  • OOSE (Object-Oriented Software Engineering) by Ivar Jacobson.
  • Odell's methodology.
  • Shlaer and Mellor methodology.

UML was adopted by the Object Management Group (OMG) as a de facto standard in 1997. OMG, an association of industries, promotes consensus on notations and techniques to foster widespread adoption and standardization.

What is a Model?

A model is an abstraction of a system that captures essential aspects while omitting or simplifying others. Models in software development can be graphical, textual, mathematical, or code-based. UML is primarily a graphical modeling tool, often accompanied by textual explanations for clarity.

Models are crucial for understanding complex systems, communicating designs, analyzing requirements, and facilitating the generation of test cases and even code templates (with the help of CASE tools like Rational Rose).

7.3 UML Diagrams & Views

UML provides nine standard types of diagrams (this number has evolved in later UML versions, but we'll focus on the classical set) to capture different views or perspectives of a system, much like architectural blueprints show different aspects of a building.

UML diagrams can capture the following views:

  • Users' View: Defines functionalities (facilities) made available to users. Captures the external view (black-box) of the system. (Primarily: Use Case Diagram)
  • Structural View (Static View): Defines the kinds of objects (classes) important to the system and their relationships. This structure does not change with time. (Primarily: Class Diagram, Object Diagram)
  • Behavioural View (Dynamic View): Captures how objects interact with each other over time to realize system behavior. (Primarily: Sequence Diagram, Collaboration Diagram, Statechart Diagram, Activity Diagram)
  • Implementation View: Captures important components of the system and their dependencies. (Primarily: Component Diagram)
  • Environmental View (Deployment View): Models how different components are implemented on different pieces of hardware. (Primarily: Deployment Diagram)

It's not necessary to construct all views using all diagrams for every system. The choice depends on the complexity and nature of the problem. The Users' View (Use Case Model) is often considered central, with other views conforming to it.

7.4 Use Case Model

A use case model consists of a set of "use cases," which represent the different ways users can interact with or use the system. To identify use cases, ask: "What can users do using the system?" (e.g., `issue-book`, `query-book` for a Library Information System).

Use cases partition system behavior into transactions, each performing a useful action from the user's perspective, possibly involving multiple message exchanges.

Representation of Use Cases:

Use cases are represented by drawing a Use Case Diagram and writing accompanying text descriptions.

  • Use Case Diagram:
    • Each use case is an ellipse with its name inside.
    • All use cases are enclosed in a rectangle (system boundary).
    • System name appears inside the rectangle.
    • Users are represented by stick-person icons called **Actors**. An actor represents a role played by a user or an external system. The line connecting an actor to a use case is a communication relationship.
    • **Stereotypes:** UML allows annotating basic symbols to slightly change their meaning. For example, <> can stereotype an actor representing another system.

Text Description for Use Cases:

Each use case ellipse should be accompanied by a detailed text description covering:

  • **Mainline Sequence:** The normal, successful flow of events.
  • **Alternate Scenarios/Variations:** Different paths or outcomes.
  • **Exceptional Conditions/Error Situations:** How errors are handled.
  • **Pre-conditions:** State of the system before the use case starts.
  • **Post-conditions:** State of the system after successful completion.
  • **Non-functional Requirements:** Constraints like performance, response time.
  • **Sample Dialogs:** Examples illustrating user interface interactions.
  • **Document References:** Links to relevant domain documents.
  • **Contact Persons:** Stakeholders involved in defining the use case.
Example Text Description Snippet (for Supermarket - U1: register-customer):
Mainline sequence:
1. Customer: select register customer option
2. System: display prompt to enter name, address, telephone number
3. Customer: enter the necessary values
4. System: display the generated id and success message.

Scenario 2 (Alternate): At step 4 of mainline.
4. System: displays message "customer has already registered".

Why Develop Use Case Diagrams?

  • Serve as a form of requirements specification.
  • Form the core model to which other models conform.
  • Help in selecting/implementing security mechanisms by identifying actor entitlements.
  • Useful for preparing user manuals targeted to different user roles (actors).
  • Actors help in identifying use cases and understanding system functioning.

Factoring Commonality Among Use Cases:

Complex use cases can be simplified, and common behavior across different use cases can be factored out for reusability (e.g., error handling). UML provides mechanisms like:

  • Generalization: A child use case inherits and specializes the behavior of a parent use case. (e.g., "Pay through credit card" and "Pay through library pay card" generalize to "Pay membership fee").
  • Include (<>): One use case explicitly includes the behavior of another at a specified point. Useful for reusing common functionality.
  • Extend (<>): One use case adds behavior to another (base) use case at defined extension points, typically for optional or exceptional behavior.

Use Case Packaging:

Use cases can be grouped into packages to organize large models, often based on related functionality or subsystems.

7.5 Class Diagrams

A class diagram describes the static structure of a system: its classes, attributes, operations, and the relationships between classes. It shows how a system is structured rather than how it behaves dynamically.

Classes:

Represent entities with common features (attributes and operations). Shown as solid outline rectangles, often with compartments for name, attributes, and operations. Class names are typically singular nouns, mixed case starting with an uppercase letter (e.g., `LibraryMember`).

Attributes:

Named properties of a class, representing data an object can hold. Listed with names, optionally type, initial value, and constraints. E.g., memberName: String, booksIssued[0..*]: Book. Multiplicity (e.g., `[10]`) indicates the number of attributes per instance.

Operations (Methods):

Functions or procedures that objects of the class can perform. Listed with names, optionally parameters (with kind: `in`, `out`, `inout`), and return type. E.g., issueBook(in bookID: String): Boolean. Abstract operations are often italicized. Class-scope operations (static) are underlined.

Relationships Between Classes:

Association:

A connection between classes, representing a structural relationship. Links are instances of associations. Shown as a solid line. Can have a name, role names at ends, and multiplicity (e.g., `1`, `0..*`, `1..5`).

Aggregation ("has-a" or "part-of"):

A special type of association representing a whole-part relationship. The aggregate (whole) is responsible for its parts. Shown with a hollow diamond at the aggregate end.

Composition (Strong Aggregation):

A stricter form of aggregation where parts are existence-dependent on the whole. If the whole is destroyed, its parts are also destroyed. Shown with a filled diamond at the composite-end.

Inheritance (Generalization/Specialization "is-a"):

Relates a superclass (general) to its subclasses (specific). Subclasses inherit attributes and operations. Shown as an empty arrowhead pointing to the superclass. Discriminators (e.g., `issuable`, `reference`) can differentiate subclasses.

Dependency:

Shows that a change in an independent class may require a change in a dependent class (e.g., one class uses another's methods or interface). Shown as a dotted arrow pointing to the independent class.

Constraints:

Conditions or integrity rules applied to model elements (e.g., permissible values for attributes, pre/post-conditions for operations). Enclosed in braces `{}`. Can be informal English or formal Object Constraint Language (OCL).

Object Diagrams

An object diagram shows a snapshot of the objects in a system (instances of classes) and their links (instances of associations) at a particular point in time. It's often called an instance diagram. Objects are typically drawn using rounded rectangles. Object diagrams are useful for explaining the dynamic working of the system and can change as execution proceeds.

Example: An object diagram might show specific LibraryMember objects like johnDoe:LibraryMember linked to specific book123:Book objects at a moment when John has borrowed that book.
