Topic 4: SQL (Structured Query Language)

SQL is the standard language for relational database management systems. Learn its fundamental syntax, different command categories, and how to use it for defining, manipulating, and controlling data.

Introduction to SQL

Query Language: A language in which a user requests information from the database. These languages are usually on a level higher than that of a standard programming language. Query languages can be procedural (user instructs the system on the sequence of operations) or non-procedural (user describes the desired information without specifying the procedure).
SQL (Structured Query Language): A standard, non-procedural query language commonly used with all relational databases for data definition and manipulation. It specifies *what* is to be done rather than *how* it is to be done. Being a higher-level language, it's closer to English and very user-friendly. ANSI has designed standard versions of SQL.
Some Important Features of SQL:
  • It is a non-procedural language.
  • It is an English-like language.
  • It can process a single record as well as sets of records at a time.
  • It is different from a third-generation language (C & COBOL). All SQL statements define what is to be done rather than how it is to be done.
  • SQL is a data sub-language consisting of three built-in languages: Data Definition Language (DDL), Data Manipulation Language (DML), and Data Control Language (DCL). (Also includes TCL and DQL).
  • It insulates the user from the underlying structure and algorithm.
  • SQL has facilities for defining database views, security, integrity constraints, transaction controls, etc.

4.1 SQL Basics (Syntax, Data Types)

SQL Syntax

SQL is followed by a unique set of rules and guidelines called Syntax. All SQL statements start with keywords like SELECT, INSERT, UPDATE, DELETE, ALTER, DROP, CREATE, USE, SHOW and all statements end with a semicolon (;).

Important Note: SQL is generally case-insensitive for keywords (e.g., SELECT and select are the same), but table and column names can be case-sensitive depending on the specific RDBMS (e.g., MySQL on some operating systems).

SQL data types specify the type of data for any object, column, variable, or expression. You choose a particular data type for a table column based on your requirements.

SQL Commands Categories

SQL commands are instructions used to communicate with the database to perform specific tasks, functions, and queries.

There are five main types of SQL commands:
  • DDL (Data Definition Language): Defines or modifies the database structure or schema.
  • DML (Data Manipulation Language): Used for accessing and manipulating data in existing schema objects.
  • DCL (Data Control Language): Deals with rights, permissions, and other controls of the database system.
  • TCL (Transaction Control Language): Manages transactions in the database.
  • DQL (Data Query Language): Used to fetch data from the database. (Often considered part of DML).

4.2 Data Definition Language (DDL)

DDL commands define a set of commands used in the creation and modification of schema objects such as tables, indexes, and views. These commands provide the ability to create, alter, and drop these objects. They are related to the management and administration of databases. Before and after each DDL statement, current transactions are implicitly committed (changes are permanently stored).

Common DDL Commands:
  • CREATE: Used to create new database objects (e.g., CREATE TABLE, CREATE DATABASE, CREATE INDEX, CREATE VIEW).
    CREATE TABLE table_name ( column1 datatype constraint, column2 datatype constraint, ... columnN datatype constraint, PRIMARY KEY (one or more columns) );
    CREATE TABLE Student (Rollno VARCHAR(20), Name VARCHAR(30), Address VARCHAR(50));
  • ALTER: Modifies the structure of an existing database object (e.g., ALTER TABLE to add/drop/modify columns or constraints).
    ALTER TABLE table_name ADD column_name datatype; ALTER TABLE table_name DROP COLUMN column_name; ALTER TABLE table_name MODIFY COLUMN column_name new_datatype; /* Syntax varies by RDBMS */ ALTER TABLE table_name RENAME TO new_table_name; /* Syntax varies */
    ALTER TABLE Student ADD Dob DATE;
  • DROP: Deletes an existing database object (e.g., DROP TABLE, DROP DATABASE, DROP INDEX).
    DROP TABLE table_name;
    DROP TABLE Student;
  • TRUNCATE: Removes all records from a table, but the table structure remains. This operation is usually faster than DELETE without a WHERE clause and often cannot be rolled back.
    TRUNCATE TABLE table_name;
  • RENAME: Renames an existing database object (often part of ALTER or a separate command).
  • DESC / DESCRIBE: (Not standard SQL, but common) Shows the structure of a table.
    DESC table_name;

4.3 Data Manipulation Language (DML)

DML commands are used to query and modify data within existing schema objects. Changes made by DML commands are not permanent until explicitly committed (unless auto-commit is enabled).

Common DML Commands:
  • SELECT: Retrieves data from one or more tables. This is also often categorized under DQL (Data Query Language).
    SELECT column1, column2, ...columnN FROM table_name [WHERE condition] [GROUP BY column_name] [HAVING condition] [ORDER BY column_name [ASC|DESC]];
    SELECT Rollno, Name FROM Student WHERE Address = 'Delhi';

    Common clauses used with SELECT:

    • DISTINCT: Returns only unique values for specified columns.
    • WHERE: Filters records based on a condition.
    • AND / OR: Combines multiple conditions in a WHERE clause.
    • IN: Specifies multiple possible values for a column.
    • BETWEEN: Selects values within a given range.
    • LIKE: Searches for a specified pattern in a column (using wildcards like % and _).
    • ORDER BY: Sorts the result set in ascending (ASC) or descending (DESC) order.
    • GROUP BY: Groups rows that have the same values in specified columns into summary rows, often used with aggregate functions.
    • HAVING: Filters groups created by GROUP BY (similar to WHERE but for groups).
    • COUNT: (Aggregate function) Returns the number of rows.
  • INSERT: Adds new rows (records) into a table.
    INSERT INTO table_name (column1, column2, ..., columnN) VALUES (value1, value2, ..., valueN);
    INSERT INTO Student (Rollno, Name) VALUES ('S101', 'Alice');
  • UPDATE: Modifies existing records in a table.
    UPDATE table_name SET column1 = value1, column2 = value2, ... WHERE condition;
    UPDATE Student SET Address = 'Mumbai' WHERE Rollno = 'S101';
  • DELETE: Removes existing records from a table.
    DELETE FROM table_name WHERE condition;
    DELETE FROM Student WHERE Rollno = 'S102';

4.4 Data Control Language (DCL)

DCL commands are used to control access to data and database objects, typically dealing with user permissions and rights.

Common DCL Commands:
  • GRANT: Gives users access privileges to database objects.
    GRANT privilege_list ON object_name TO user_name; GRANT CREATE SESSION TO user_name; /* System privilege example */ GRANT SELECT, INSERT ON TableName TO user_name; /* Object privilege example */
    GRANT SELECT ON Student TO 'user1';
  • REVOKE: Withdraws user access privileges previously granted.
    REVOKE privilege_list ON object_name FROM user_name;
    REVOKE SELECT ON Student FROM 'user1';
  • CREATE USER: (Often considered DCL or part of user management) Creates a new database user.
    CREATE USER 'username'@'host' IDENTIFIED BY 'password';
  • DROP USER: Deletes a database user.

Transaction Control Language (TCL)

TCL commands are used to manage transactions in the database. Transactions are sequences of one or more SQL operations executed as a single logical unit of work. TCL commands can only be used with DML commands like INSERT, DELETE, and UPDATE.

Common TCL Commands:
  • COMMIT: Saves all transactions to the database permanently.
    COMMIT;
  • ROLLBACK: Undoes transactions that have not already been saved (committed) to the database.
    ROLLBACK;
  • SAVEPOINT: Sets a point within a transaction to which you can later roll back.
    SAVEPOINT savepoint_name;
    To roll back to a savepoint: ROLLBACK TO SAVEPOINT savepoint_name;

Data Query Language (DQL)

DQL is used to fetch data from the database. The primary command for DQL is SELECT. While often grouped under DML, some classifications treat it separately due to its distinct purpose of querying rather than modifying data.

SELECT Statement (Recap):
SELECT [DISTINCT] column1, column2, ... FROM table_name [WHERE condition] [GROUP BY column_list] [HAVING condition] [ORDER BY column_list [ASC|DESC]] [INTO new_table];

SELECT * FROM table_name; retrieves all columns from the table.

SELECT TOP N * FROM table_name; (or LIMIT N in some RDBMS) retrieves the top N records.

4.5 Constraints in SQL

Constraints are rules enforced on data columns in a table. They are used to limit the type of data that can go into a table, ensuring accuracy and reliability. Constraints can be column-level (applied to a single column) or table-level (applied to the whole table or multiple columns).

Commonly Used Constraints:
  • NOT NULL: Ensures that a column cannot have a NULL value.
  • UNIQUE: Ensures that all values in a column (or a set of columns) are different. Allows one NULL value.
  • PRIMARY KEY: Uniquely identifies each row/record in a database table. It must contain unique values and cannot have NULL values. A table can have only one primary key.
    -- During table creation CREATE TABLE Customers ( ID INT NOT NULL, Name VARCHAR(20) NOT NULL, PRIMARY KEY (ID) ); -- Adding to existing table ALTER TABLE Customers ADD PRIMARY KEY (ID);
  • FOREIGN KEY: Uniquely identifies a row/record in another table (or sometimes the same table). It is used to link two tables together and enforces referential integrity.
    -- During table creation CREATE TABLE Orders ( OrderID INT NOT NULL, CustomerID INT, OrderDate DATE, PRIMARY KEY (OrderID), FOREIGN KEY (CustomerID) REFERENCES Customers(ID) ); -- Adding to existing table ALTER TABLE Orders ADD FOREIGN KEY (CustomerID) REFERENCES Customers(ID);
  • CHECK: Ensures that all values in a column satisfy a specific condition.
    CREATE TABLE Customers ( ID INT NOT NULL, Age INT, PRIMARY KEY (ID), CHECK (Age >= 18) );
  • DEFAULT: Provides a default value for a column when no value is specified during an INSERT.
  • INDEX: Used to create and retrieve data from the database very quickly (covered in more detail later).

4.6 Aggregate Functions

Aggregate functions perform a calculation on a set of values and return a single summary value. They are often used with the GROUP BY clause.

Common Aggregate Functions:
  • COUNT([DISTINCT] expression | *): Returns the number of rows or non-NULL values.
  • SUM([DISTINCT] expression): Returns the total sum of a numeric column.
  • AVG([DISTINCT] expression): Returns the average value of a numeric column.
  • MIN(expression): Returns the minimum value in a column.
  • MAX(expression): Returns the maximum value in a column.
SELECT COUNT(*) FROM Employees; SELECT AVG(Salary) FROM Employees WHERE Department = 'Sales'; SELECT Department, SUM(Salary) FROM Employees GROUP BY Department;

4.7 Joins

A SQL JOIN clause is used to combine rows from two or more tables based on a related column between them. Several types of joins are available:

  • INNER JOIN (or JOIN): Returns rows when there is a match in both tables. This is the most common type of join.
    SELECT table1.column1, table2.column2... FROM table1 INNER JOIN table2 ON table1.common_field = table2.common_field;
  • LEFT JOIN (or LEFT OUTER JOIN): Returns all rows from the left table, and the matched rows from the right table. If there is no match, NULLs are returned for columns from the right table.
    SELECT table1.column1, table2.column2... FROM table1 LEFT JOIN table2 ON table1.common_field = table2.common_field;
  • RIGHT JOIN (or RIGHT OUTER JOIN): Returns all rows from the right table, and the matched rows from the left table. If there is no match, NULLs are returned for columns from the left table.
    SELECT table1.column1, table2.column2... FROM table1 RIGHT JOIN table2 ON table1.common_field = table2.common_field;
  • FULL JOIN (or FULL OUTER JOIN): Returns all rows when there is a match in either the left or right table. If there's no match, NULLs are returned for the columns of the table without a match.
    SELECT table1.column1, table2.column2... FROM table1 FULL OUTER JOIN table2 ON table1.common_field = table2.common_field;
  • SELF JOIN: Used to join a table to itself, as if the table were two tables, temporarily renaming at least one table in the SQL statement.
    SELECT a.column_name, b.column_name... FROM table1 a, table1 b WHERE a.common_field = b.other_field;
  • CROSS JOIN (or CARTESIAN JOIN): Returns the Cartesian product of the sets of records from the two or more joined tables (all possible combinations of rows). Generally used with caution.
    SELECT table1.column1, table2.column2... FROM table1 CROSS JOIN table2;

4.8 Nested and Correlated Subqueries

Subquery (or Inner Query / Nested Query): A query embedded within another SQL query (the outer query). It is enclosed in parentheses and typically executed first. The result of the subquery is then used by the outer query.
Types based on execution:
  • Non-correlated (Simple) Subquery: The inner query is executed once, and its result is used by the outer query. The inner query is independent of the outer query. Find employees whose salary is greater than the average salary: 
    SELECT Name FROM Employees WHERE Salary > (SELECT AVG(Salary) FROM Employees);
  • Correlated Subquery: The inner query depends on the outer query for its values. The inner query is executed repeatedly, once for each row processed by the outer query. Often uses operators like EXISTS or NOT EXISTS. Find employees who have at least one dependent: 
    SELECT E.Name
FROM Employee E
WHERE EXISTS (SELECT * FROM Dependent D WHERE D.EmpID = E.EmpID);
Usage in Clauses:

Subqueries can be used in various parts of an SQL statement:

  • In the SELECT list (scalar subquery - must return a single value).
  • In the FROM clause (derived table).
  • In the WHERE clause (most common, for filtering).
  • In the HAVING clause.
Operators with Subqueries:
  • IN / NOT IN: Checks if a value matches any value in the result set of the subquery.
  • ANY / SOME: Compares a value to each value returned by the subquery (e.g., > ANY).
  • ALL: Compares a value to all values returned by the subquery (e.g., > ALL).
  • EXISTS / NOT EXISTS: Checks if the subquery returns any rows.

4.9 Views, Indexes, Triggers, Stored Procedures

Views

A view is a virtual table whose contents are defined by a query. It's a logical subset of data from one or more tables. Views do not store data themselves (usually, except for materialized views in some RDBMS); they are stored as a query definition in the database.

Advantages of Views:
  • Security: Restrict access to specific data by showing users only relevant columns or rows.
  • Simplicity: Simplify complex queries by presenting data from multiple tables as a single virtual table.
  • Data Independence: Shield applications from changes in the underlying base table structures.
  • Consistency: Ensure data is presented in a consistent manner to different users.
Types of Views:
  • Simple Views: Based on a single table, do not contain functions, groups of data, or DISTINCT. Often updatable.
  • Complex Views: Based on multiple tables, may contain functions, GROUP BY, DISTINCT, etc. Usually not updatable directly.
CREATE VIEW view_name AS SELECT column1, column2, ... FROM table_name WHERE condition; DROP VIEW view_name;
CREATE VIEW HighEarners AS SELECT EmpName, Salary FROM Employees WHERE Salary > 70000;

Indexes

Indexes are special lookup tables that the database search engine can use to speed up data retrieval. An index is a pointer to data in a table, similar to an index in the back of a book. Indexes speed up SELECT queries and WHERE clauses but can slow down data input (INSERT, UPDATE, DELETE operations) as indexes also need to be updated.

Types of Indexes:
  • Single-Column Index: Created on a single column.
  • Composite Index (Multi-Column Index): Created on two or more columns.
  • Unique Index: Ensures that the indexed column(s) contain only unique values. Primary keys automatically have unique indexes.
  • Implicit Indexes: Automatically created by the database server for primary key and unique constraints.
CREATE [UNIQUE] INDEX index_name ON table_name (column1, column2, ...); DROP INDEX index_name ON table_name; /* Syntax varies by RDBMS */
When to avoid indexes:
  • On small tables.
  • On tables with frequent, large batch update or insert operations.
  • On columns that contain a high number of NULL values.
  • On columns that are frequently manipulated.

Triggers

A trigger is a stored procedure in a database that automatically invokes (fires) when a specified event (e.g., INSERT, UPDATE, DELETE) occurs on a particular table or view. Triggers are used to maintain data integrity, enforce complex business rules, or automate tasks.

Triggers can be defined to execute BEFORE or AFTER the triggering event, and can be row-level (fires for each affected row) or statement-level (fires once per SQL statement).

CREATE TRIGGER trigger_name {BEFORE | AFTER} {INSERT | UPDATE | DELETE} ON table_name [FOR EACH ROW] -- For row-level trigger BEGIN -- Trigger body: SQL statements END;
A trigger to automatically update an `audit_log` table whenever a record in the `Employees` table is modified.

Stored Procedures

A stored procedure is a precompiled collection of one or more SQL statements and optional control-of-flow statements, stored in the database. Stored procedures can accept input parameters and return output parameters or result sets.

Advantages of Stored Procedures:
  • Performance: Precompiled, so execution is faster.
  • Reduced Network Traffic: Instead of sending multiple SQL statements, applications send a single call to execute the procedure.
  • Reusability & Modularity: Can be called by multiple applications or users.
  • Security: Can grant execute permissions on procedures without granting direct table access.
  • Encapsulation of Business Logic: Complex business rules can be embedded within procedures.
CREATE PROCEDURE procedure_name ( @parameter1 datatype, @parameter2 datatype OUTPUT -- Example of an output parameter ) AS BEGIN -- SQL statements and logic SELECT ... ; SET @parameter2 = ... ; END; -- To execute: EXECUTE procedure_name @parameter1_value, @output_variable OUTPUT;
A stored procedure `AddCustomer` that takes customer details as input and inserts a new record into the `Customers` table.

SET NOCOUNT ON/OFF: In Transact-SQL (SQL Server), SET NOCOUNT ON prevents the sending of "done_in_proc" messages (indicating rows affected) to the client for each statement in a stored procedure. This can improve performance by reducing network traffic, especially for procedures with loops or many statements.
