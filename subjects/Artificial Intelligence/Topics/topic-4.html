    Introduction to Game Playing in AI

    Game playing has always been a fascinating domain for artificial intelligence (AI). From the early days of computer science to the current era of advanced deep learning systems, games have served as benchmarks for AI development. They offer structured environments with clear rules, making them ideal for training algorithms to solve complex problems. With AI's ability to learn, adapt, and make strategic decisions, it is now becoming an essential player in various gaming domains, reshaping how we experience and interact with games.

    Game Playing is an important domain of artificial intelligence. Games don't require much knowledge; the only knowledge we need to provide is the rules, legal moves and the conditions of winning or losing the game. Both players try to win the game. So, both of them try to make the best move possible at each turn. Searching techniques like BFS (Breadth First Search) are not accurate for this as the branching factor is very high, so searching will take a lot of time. Game playing in AI is an active area of research and has many practical applications, including game development, education, and military training.

    Types of Games in AI

    Games can be classified based on several characteristics, which influence the strategies AI uses:

    • Perfect vs. Imperfect Information:
      • Perfect Information: All players can see the exact state of the game. Examples: Chess, Checkers, Go, Othello, Backgammon.
      • Imperfect Information: Some information is hidden from players. Examples: Scrabble, Bridge, Poker, most multi-player video games.
    • Deterministic vs. Stochastic:
      • Deterministic: The outcome of an action is fully determined by the choice of action. Example: Chess.
      • Stochastic: The outcome of an action is partially determined by chance (e.g., dice rolls, card shuffles). Examples: Backgammon, Poker.

    This topic primarily focuses on two-player, perfect information, deterministic, zero-sum games (where one player's gain is the other's loss).

    Historical Milestones: Human vs. Computer

    AI has achieved remarkable success in various games, often surpassing human champions:

    Key Milestones
    • Checkers: 1994 - Chinook (UofA) beat human world champion Marion Tinsley.
    • Chess: 1997 - IBM's Deep Blue beat world champion Garry Kasparov. 2002 - Fritz drew with Vladimir Kramnik.
    • Othello: 1997 - Logistello (NEC Research) beat world champion Takeshi Murakami. Today, human champions often refuse to play top computer programs.
    • Go: Master-level play achieved by AI (e.g., AlphaGo by DeepMind in 2016 beating Lee Sedol), a game once thought too complex for AI to master due to its vast search space.
    • Scrabble: 1998 - Maven (UofA) beats world champion Adam Logan.
    • Poker: 2008 - Polaris (UofA) beats some top online human players. More recent AIs like Pluribus and Libratus have shown superhuman performance in multi-player no-limit Texas Hold'em.
    • Bridge: 1988 - Ginsberg's program placed 12th in world championships. Coordination remains difficult.
    • Commercial, Multi-player Games (e.g., Dota 2, StarCraft II): Very hard problems with real-time constraints, dynamic opponents, and imperfect information. Progress is being made, often with the goal of creating "interesting" AI opponents rather than just defeating humans.

    Game Playing as Search

    We can formulate two-player, perfect information, deterministic games as search problems:

    • State: The current configuration of the board/game.
    • Operators (Actions): Legal moves available to the current player.
    • Initial State: The starting configuration of the game.
    • Terminal States (Goal): States where the game is won, lost, or drawn.
    • Utility (Cost/Payoff): A function that assigns a numerical value to terminal states. For example: +1 for winning, -1 for losing, 0 for a draw. More complex games might use points, money, etc.

    The goal is to find a strategy (a way of picking moves) that maximizes the AI's utility, assuming the opponent plays optimally to minimize the AI's utility.

    The Adversarial Search Challenge

    Game playing search is different from standard search problems because of the presence of a malicious opponent.

    • The opponent is trying to make things good for itself and bad for our AI.
    • We have to simulate the opponent's decision-making process.

    The main idea is to define utility from a single agent's perspective:

    • A MAX player (our AI) who wants to maximize its utility.
    • A MIN player (the opponent) who wants to minimize MAX's utility (which is equivalent to maximizing their own utility in a zero-sum game).

    Minimax Algorithm

    The Minimax algorithm is a decision-making algorithm used in AI, game theory, and decision theory for two-player zero-sum games. It determines the optimal move for a player assuming that the opponent also plays optimally. It's a depth-first, depth-limited search procedure.

    Core Idea:

    1. Expand a complete search tree (game tree) until terminal states are reached and their utilities can be computed.
    2. Go back up from the leaves towards the current state:
      • At each MIN node (opponent's turn), back up the worst (minimum) value among its children.
      • At each MAX node (AI's turn), back up the best (maximum) value among its children.

    Minimax Search Tree Example

    Image: Generic Minimax tree with MAX/MIN levels and backed-up values (like the one with root value 3).

    Minimax Algorithm (Recursive Definition):

    MINIMAX(Node n) = 
  UTILITY(n)                             if n is a terminal state
  maxs ∈ Successors(n) MINIMAX(s)      if n is a MAX node
  mins ∈ Successors(n) MINIMAX(s)      if n is a MIN node

    Working of Minimax (Example: Tic-Tac-Toe):

    Tic-Tac-Toe is a simple game to illustrate Minimax:

    1. Start with the initial game state.
    2. Generate all possible moves for the current player.
    3. Evaluate each move by simulating the resulting game state and assigning a score (utility) using Minimax.
    4. If the game is over (win, lose, draw), assign a utility score accordingly (+1 for MAX win, -1 for MIN win/MAX loss, 0 for draw).
    5. If a maximum depth is reached (for complex games) or the game is over, return the score.
    6. If it's MAX's turn, select the move leading to the highest Minimax value.
    7. If it's MIN's turn, select the move leading to the lowest Minimax value.
    8. Repeat steps until the game is over or a decision is made.

    Properties of Minimax Search
    • Completeness: Yes, if the game tree is finite.
    • Optimality: Yes, against an optimal opponent. If the opponent doesn't play optimally, Minimax might still find a good move, but not necessarily the absolute best against that specific sub-optimal play.
    • Time Complexity: O(bm), where 'b' is the branching factor (number of legal moves per state) and 'm' is the maximum depth of the game tree.
    • Space Complexity: O(bm) for a depth-first exploration (stores nodes along the current path).

    Limitations of Minimax:

    The main drawback is its high time complexity. For games like Chess (b ≈ 35, m ≈ 100 for "reasonable" games), an exact solution using pure Minimax is impossible. This leads to the need for:

    • Depth Limits (Cutoff): Stop searching after a certain depth.
    • Evaluation Functions: Estimate the utility of non-terminal states.
    • Pruning Techniques: Like Alpha-Beta Pruning.

    Evaluation Functions:

    For games where the terminal states are too deep to reach, we use an evaluation function to estimate the expected utility of a game from a given position.
    Example for Chess: Assign values to pieces (Pawn=1, Knight=3, etc.). Score = (Sum of AI's pieces) - (Sum of opponent's pieces).
    More sophisticated functions use weighted features: w1f1(s) + w2f2(s) + ... (e.g., material balance, mobility, king safety).

    Evaluation functions are often approximate and more accurate near the end of the game. Only the order of the numbers matters for Minimax decision if the transformation is monotonic.

    Alpha-Beta Pruning

    Alpha-Beta Pruning is an optimization technique for the Minimax algorithm. It reduces the number of nodes evaluated in the search tree by "pruning" branches that cannot influence the final decision. It returns the same move as Minimax would, but by examining fewer states.

    Core Idea:

    Keep track of two values during the search:

    • Alpha (α): The best (highest-value) choice found so far along the path for MAX. Initialized to -∞.
    • Beta (β): The best (lowest-value) choice found so far along the path for MIN. Initialized to +∞.

    Pruning occurs when the current value of a node being examined is worse than the current α or β value for its ancestors. Specifically:

    • If a MIN node's value becomes less than or equal to α (MAX's current best guaranteed score), MAX will never choose this path, so further exploration of this MIN node's children is unnecessary. (Prune below this MIN node: β ≤ α)
    • If a MAX node's value becomes greater than or equal to β (MIN's current best guaranteed score for MAX), MIN will never allow this path, so further exploration of this MAX node's children is unnecessary. (Prune below this MAX node: α ≥ β)
    The main condition for Alpha-Beta pruning is: α ≥ β.

    Image: Game tree showing how Alpha-Beta values propagate and where pruning occurs (like the example with a branch crossed out).
    Alpha-Beta Algorithm (Conceptual Pseudocode)

    Instead of a single MinimaxValue function, Alpha-Beta uses two mutually recursive functions, MaxValue and MinValue, each taking the current state, alpha, and beta as parameters.

    // MaxValue function for MAX player
function MaxValue(state, alpha, beta):
  if CutoffTest(state) or TerminalTest(state):
    return Evaluation(state)
  v = -infinity
  for each successor s of state:
    v = max(v, MinValue(s, alpha, beta))
    if v >= beta: // Pruning condition for MAX
      return v 
    alpha = max(alpha, v)
  return v

// MinValue function for MIN player
function MinValue(state, alpha, beta):
  if CutoffTest(state) or TerminalTest(state):
    return Evaluation(state)
  v = +infinity
  for each successor s of state:
    v = min(v, MaxValue(s, alpha, beta))
    if v <= alpha: // Pruning condition for MIN
      return v
    beta = min(beta, v)
  return v
    Properties of Alpha-Beta Pruning
    • Result: Pruning does not affect the final Minimax value returned (it yields the same result as Minimax).
    • Efficiency: The effectiveness of pruning heavily depends on move ordering.
      • With perfect ordering (best moves explored first), the time complexity can be reduced to approximately O(bm/2), effectively doubling the searchable depth.
      • With worst ordering, no pruning occurs, and complexity remains O(bm).
      • On average, with random ordering, complexity is often around O(b3m/4).
    • Demonstrates the value of reasoning about which computations are important.

    Drawbacks of Alpha-Beta Pruning:

    • If the branching factor is very large (e.g., in Go, b ≈ 300), search depth is still limited.
    • Optimal play is guaranteed against an optimal opponent if search proceeds to the end. However, if heuristics are used or the opponent is not optimal, the assumptions might not hold perfectly.

    Game Trees and Strategic Considerations

    A game tree is a directed graph whose nodes are positions in a game and whose edges are moves. The complete game tree for a game starts at the initial position and contains all possible moves from each position; the leaves are terminal positions.

    Each level of the tree is often called a ply. For example, a 2-ply search looks two moves ahead (one for MAX, one for MIN).

    Strategic considerations in AI game playing involve:

    • Choosing an appropriate search depth (cutoff): Balancing computational cost with search quality.
    • Designing effective evaluation functions: Crucial when a full search to terminal states is infeasible.
    • Move ordering for Alpha-Beta Pruning: Heuristics can be used to order moves to maximize pruning.
    • Handling uncertainty and imperfect information: Requires different techniques beyond standard Minimax (e.g., Monte Carlo Tree Search for games like Go, or modeling opponent beliefs).
