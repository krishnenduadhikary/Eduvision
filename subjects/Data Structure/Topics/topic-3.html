Topic 3

Strings

Strings are fundamental data structures in computer science, representing sequences of characters. They are used extensively in various applications, from storing simple text like names and messages to complex data like source code, web pages, and genetic sequences. Understanding string representations, operations, and algorithms for pattern matching is crucial for any programmer.

Fig: Strings are used to represent textual data in various forms.

1. String Representations

How strings are stored in memory can vary, impacting performance, safety, and ease of use. The primary ways include C-style strings, general character arrays, and string objects.

a. C-style Strings (Null-terminated)

In C and C++, a string is traditionally represented as an array of characters ending with a special null character (\0). This null terminator signals the end of the string.

Fig: C-style string "HELLO" stored with a null terminator.

Characteristics:

  • Memory: Stored contiguously. Length is determined by scanning for \0.
  • Manipulation: Requires manual memory management. Functions from (like strcpy, strlen) are used.
  • Pros: Memory efficient, direct memory access.
  • Cons: Prone to buffer overflows, finding length can be slow (O(n)). 
// Example of a C-style string
#include 
#include 

int main() {
    char greeting[6] = {'H', 'e', 'l', 'l', 'o', '\0'};
    char another_greeting[] = "World"; // Compiler adds \0 automatically

    printf("Greeting: %s\n", greeting);
    printf("Length of another_greeting: %zu\n", strlen(another_greeting));
    return 0;
}

b. Character Arrays (General)

Some languages or specific use cases might use character arrays that are not necessarily null-terminated. The length might be stored separately or implied by the array's fixed size. This is less common for general-purpose string manipulation but can be found in fixed-buffer scenarios.

c. String Objects/Classes

Modern languages (Java, Python, C++ with std::string, C#) provide string objects or classes. These are higher-level abstractions that manage memory automatically and offer a rich set of built-in methods.

Fig: Conceptual model of a string object storing data, length, and possibly capacity.

Characteristics:

  • Memory Management: Handled internally; strings can often grow or shrink dynamically.
  • Functionality: Include methods for length, concatenation, substring, search, etc. Length is typically stored, making access O(1).
  • Pros: Safer (reduced risk of buffer overflows), easier to use, more features.
  • Cons: Can have more memory overhead, operations might involve hidden allocations.
# Example of string objects in Python
my_string = "Hello, EduVision!"
another_string = " Students"

# Concatenation
combined_string = my_string + another_string
print(combined_string)

# Length
print(f"Length: {len(combined_string)}")

# Substring (slicing)
substring = my_string[7:16] # Extracts "EduVision"
print(f"Substring: {substring}")

2. Common String Operations

Regardless of representation, several operations are commonly performed on strings:

Fig: Visual examples of common string operations.

Comparison

Determines if two strings are identical or their lexicographical order (e.g., "apple" < "banana").

Concatenation

Joins two or more strings to form a new string (e.g., "Hello" + " " + "World" = "Hello World").

Substring

Extracts a portion of a string (e.g., substring of "EduVision" from index 3 of length 3 is "Vis").

Length/Size

Finds the number of characters in a string.

Search (Find)

Locates the first occurrence of a character or a substring within another string.

Modification

Includes operations like replacing characters/substrings, converting case, trimming whitespace.

3. Pattern Matching Algorithms

Pattern matching involves finding all occurrences of a given pattern string (P) within a larger text string (T). This is a classic problem with numerous applications.

a. Naive Algorithm (Brute-Force)

The Naive algorithm checks for the pattern at every possible starting position in the text. It slides the pattern one character at a time and compares characters.

Fig: The Naive algorithm slides the pattern P across text T.

Process:

  • Align pattern P with the start of text T.
  • Compare P with the corresponding segment of T, character by character.
  • If all characters match, a pattern occurrence is found.
  • Slide P one position to the right in T and repeat.

Complexity:

  • Time: O(m * (n-m+1)) or O(m*n) in the worst case.
  • Space: O(1).

When to use:

Simple to implement, efficient for small texts/patterns.

NaiveSearch(Text T, Pattern P)
  n = length(T)
  m = length(P)
  for s = 0 to n-m
    match = true
    for j = 0 to m-1
      if T[s+j] != P[j]
        match = false
        break
    if match == true
      print "Pattern found at index " + s

b. Knuth-Morris-Pratt (KMP) Algorithm

The KMP algorithm improves by using information from previous matches via an LPS (Longest Proper Prefix which is also Suffix) array to avoid redundant comparisons.

Fig: KMP uses an LPS array for intelligent shifts.

LPS Array:

lps[i] stores the length of the longest proper prefix of P[0...i] that is also a suffix of P[0...i].

Process:

  • Preprocessing: Compute the LPS array for P (O(m)).
  • Searching: On mismatch at P[j], shift P using lps[j-1].

Complexity:

  • Time: O(n + m).
  • Space: O(m) for LPS array.

When to use:

General-purpose string searching, effective with repeating sub-patterns.

c. Rabin-Karp Algorithm

Rabin-Karp uses hashing. It calculates hash values for the pattern and text substrings. Matching hashes trigger character-by-character verification.

Fig: Rabin-Karp compares hash values, using a rolling hash.

Rolling Hash:

Efficiently computes the hash of the next substring from the current one in O(1) time.

Process:

  • Calculate hash(P).
  • Calculate hash for the first m-length segment of T.
  • Compare hashes. If equal, verify.
  • "Roll" hash to the next segment and repeat.

Complexity:

  • Time: Average O(n + m). Worst Case O(nm).
  • Space: O(1).

When to use:

Multiple pattern matching, plagiarism detection, large alphabets.

4. Summary of Pattern Matching Algorithms

Choosing the right algorithm depends on factors like text/pattern length, number of patterns, and data nature.

Algorithm Preprocessing Matching (Worst) Space Key Idea
Naive None (O(1)) O(m(n-m+1)) O(1) Simple character-by-character sliding window.
KMP O(m) O(n) O(m) Uses LPS array to skip redundant comparisons.
Rabin-Karp O(m) (hash) O(nm) (Avg O(n+m)) O(1) Hashing and rolling hash for quick comparisons.
